# V2 整改任务总台账

## 1. 说明

本台账用于统一管理 V2 阶段的整改任务，避免后续在任务下发、回传、验收、关单过程中出现编号混乱、状态不清和结论不可追溯的问题。

后续所有整改任务，统一按“任务编号 + 名称 + 状态 + 验收结论”方式管理。

## 2. 任务编号规则

- `R`：规则与风险识别整改
- `E`：样本、评测、监督标准整改
- `W`：Web、链路、展示与运行整改

## 3. 状态定义

- `待下发`：任务已明确，但尚未正式交付给 T
- `整改中`：T 已开始整改，尚未提交验收
- `待验收`：T 已回传结果，等待 M 复核
- `已通过`：M 已完成验收并判定通过
- `未通过`：M 已验收但未过线，需要继续整改
- `已关闭`：任务通过验收，且确认不再继续跟进

### R-002

- 任务名称：补齐“强制性标准条款未按评审规则加注★”风险识别
- 任务类型：规则整改
- 当前状态：`已关闭`
- 当前结论：已完成验收并正式关单

#### 风险来源

- 规则要求：`含有GB（不含GB/T）或国家强制性标准的描述中需含有★号`
- 业务条款示例：`符合 GB252 或 BS2869 标准`
- 正文未加 `★`

#### 整改目标

- 识别 `GB` 非 `GB/T`
- 识别“规则要求必须加 `★`”
- 对齐正文未加 `★` 的情形并报风险

#### 标准风险文案

- 风险标题：`强制性标准条款未按评审规则标注★，可能导致实质性响应边界不清`
- 问题定性：`中风险`
- 审查类型：`评审规则一致性 / 实质性条款标识完整性`

#### 风险判断要求

- 评审规则已明确：含 `GB`（不含 `GB/T`）或国家强制性标准的描述，应标注 `★`
- 当前条款引用了 `GB252` 等 `GB` 非 `GB/T` 标准，但正文未加 `★`
- 可能导致投标人无法准确识别其是否属于实质性条款
- 若评审阶段按实质性条款处理，存在废标争议和评审口径不一致风险

#### 整改建议要求

- 若该条款属于实质性要求，应在条款前明确加注 `★`
- 若不作为实质性条款，应同步修改评审规则或补充解释，保持规则与正文一致

#### 规则整改要求

1. 新增评审规则侧信号提取
2. 识别下列语义：
   - `含有GB（不含GB/T）`
   - `国家强制性标准`
   - `需含有★号`
   - `应标注★`
   - `实质性条款需加★`
3. 建议形成结构化信号：
   - `star_required_for_gb_non_t = true`
   - `star_required_for_mandatory_standard = true`

#### 正文识别要求

1. 识别正文技术条款中：
   - 是否命中 `GB` 但不是 `GB/T`
   - 是否命中“国家强制性标准”语义
   - 条款标题或条款正文前是否已有 `★`
2. 建议形成结构化信号：
   - `contains_gb_non_t = true`
   - `contains_gbt = false`
   - `contains_mandatory_standard = false/true`
   - `has_star_marker = false`

#### Cross-check 要求

当同时满足：

- 规则要求该类条款必须加 `★`
- 正文条款命中 `GB`（非 `GB/T`）或国家强制性标准
- 正文条款未加 `★`

则输出该风险点。

#### 样本整改要求

1. 正样本
   - `符合 GB252 标准`，未加 `★`
   - `符合 GB 50016 标准`，未加 `★`
   - `符合国家强制性标准`，未加 `★`
   - `符合 GBxxx 或外标`，未加 `★`
2. 负样本
   - `符合 GB/T 17626 标准`，未加 `★`，不应触发
   - `★符合 GB252 标准`，已加 `★`，不应触发
   - 规则未要求 `GB` 条款必须加 `★` 的项目，不应触发
   - 普通技术参数条款，不含 `GB`、不含强制标准语义，不应触发
3. 真实业务表达样本
   - `（7）燃油标准，采用0号轻柴油，符合 GB252 或 BS2869 标准。`
   - 配套规则：`评审规则合理性-含有GB（不含GB/T）或国家强制性标准的描述中需含有★号。`

#### 测试整改要求

1. 规则抽取测试
2. 正文识别测试
3. 端到端测试

#### 交付物要求

2. 新增了哪些结构化信号
3. Cross-check 规则怎么设计
4. 新增了哪些样本

6. 测试运行结果
