# 项目角色职责与问题闭环机制 V1

## 1. 文档目的

本文件用于明确 `test_getst` 项目在当前阶段的角色分工、协作边界与问题闭环机制，降低以下风险：

- 客户问题进入后无人负责
- 运营、管理、开发对同一问题理解不一致
- 问题被记录但没有真正关闭
- 修复动作没有进入版本计划或质量门

本机制适用于当前项目下的三类角色：

- `O-运营`
- `M-管理`
- `T-开发`


## 2. 项目定位

`test_getst` 是一套招标文件智能审查系统，当前已形成 `V1` 和 `V2` 两套能力链路，其中 `V2` 正在向“成熟版”演进。

项目目标不是单次模型直接生成结果，而是通过分层处理提升：

- 合规风险识别能力
- 结果可解释性
- 证据可追踪性
- 问题可复核与可排障性

当前 `V2` 主要链路包括：

1. 文本抽取
2. 全文基线审查
3. 结构识别与证据召回
4. 专题深审
5. 汇总分析与统一报告
6. Web 展示与结果落盘


## 3. 角色划分总则

### 3.1 O-运营

负责项目日常运行保障、客户反馈接收、问题初步分类、问题流转与结果回传。

### 3.2 M-管理

负责版本目标、优先级、任务排期、验收标准和跨角色协同管理。

### 3.3 T-开发

负责产品实现、缺陷修复、规则与模型优化、测试与回归验证。


## 4. O-运营职责清单

### 4.1 产品运行保障

- 保障 Web 和 CLI 两种使用入口可正常使用。
- 检查文件上传、任务创建、状态轮询、结果展示、历史查看是否正常。
- 关注核心产物是否生成完整，包括：
  - `extracted_text.md`
  - `baseline_review.md`
  - `review.md`
  - `final_review.md`
  - `v2_overview.json`
  - `document_map.json`
  - `topic_reviews/*.json`

### 4.2 客户反馈接收

- 统一接收客户在使用过程中的问题、建议和异常反馈。
- 确保每条反馈都有最基本上下文：
  - 文件名
  - 反馈时间
  - 使用入口
  - 实际现象
  - 期望结果
  - 是否可复现

### 4.3 问题初步分类

- 将反馈按问题性质分类为：
  - 使用问题
  - 配置问题
  - 页面问题
  - 文本抽取问题
  - 审查结果质量问题
  - 模型或规则问题
  - 需求优化建议

### 4.4 问题初步定位

- 从业务侧判断问题大致位于哪一层：
  - 文本抽取层
  - 全文基线层
  - 结构召回层
  - 专题深审层
  - 汇总展示层
- 不要求运营直接给出技术修复方案，但必须能说明问题现象和影响范围。

### 4.5 问题流转与跟进

- 将需要排期决策的问题提交给 `M-管理`。
- 将需要实现或修复的问题提交给 `T-开发`。
- 跟踪问题从创建、处理中、待验证到关闭的全过程。
- 对外统一反馈状态，避免客户重复追问仍无人回应。

### 4.6 运营数据沉淀

- 周期性整理高频客户问题。
- 汇总高频失败文件类型和高频风险漏审类型。
- 形成可供管理与开发使用的问题趋势信息。

### 4.7 市场适配推动

- 将零散客户意见转换成产品可执行的优化建议。
- 推动产品能力随着市场和客户场景变化持续调整。


## 5. M-管理职责清单

### 5.1 版本规划

- 明确 `V1`、`V2`、`V2 成熟版` 的目标与边界。
- 决定每一阶段重点投入的能力模块。

### 5.2 优先级管理

- 对运营收集的问题进行优先级划分。
- 平衡客户紧急问题、版本建设任务、技术债和质量门要求。

### 5.3 任务拆解与排期

- 将架构方案拆解为可执行任务。
- 明确每项任务的输入、输出、负责角色与验收标准。

### 5.4 质量门管理

- 管理结构层、专题层、汇总层和主回归层的质量门。
- 决定什么标准可以上线，什么标准必须继续打磨。

### 5.5 协同推进

- 协调 `O-运营` 与 `T-开发` 的工作节奏。
- 防止客户问题长期滞留在“待确认”状态。
- 对外统一口径，对内形成闭环。

### 5.6 风险控制

- 防止范围蔓延、版本失控、验收口径不清。
- 防止为了短期过样本而损害长期泛化能力。


## 6. T-开发职责清单

### 6.1 产品功能开发

- 负责审查链路、Web 页面、CLI 工具及结果落盘能力的建设与维护。
- 保证各模块在当前架构下可运行、可回归、可定位。

### 6.2 审查能力建设

- 持续完善：
  - 全文基线审查
  - 结构识别与证据召回
  - 专题深审
  - 汇总分析与统一报告

### 6.3 规则与模型优化

- 优化 prompt、规则、后处理与合并逻辑。
- 降低漏报、误报和跨专题冲突。

### 6.4 缺陷修复

- 对运营或管理流转的问题给出技术归因。
- 完成修复并提供验证结果。
- 对无法立即修复的问题给出明确原因和后续建议。

### 6.5 测试与评测维护

- 维护项目测试、评测脚本和样本集。
- 确保每次改动后执行必要回归，避免功能回退。

### 6.6 兼容性与稳定性维护

- 保持已有产物结构兼容。
- 保证失败降级、超时回退和运行日志可用于排障。


## 7. 角色协作边界

### 7.1 O-运营边界

- 负责发现问题、记录问题、推动问题，不直接替开发决定实现方案。

### 7.2 M-管理边界

- 负责定优先级、定排期、定验收，不直接替开发完成实现细节。

### 7.3 T-开发边界

- 负责实现、修复、验证，不直接替管理决定需求范围和上线策略。


## 8. 问题闭环机制

### 8.1 问题进入

问题可来自以下渠道：

- 客户反馈
- 内部测试
- 演示使用
- 历史结果复核
- 质量门回归失败

每条问题至少记录以下信息：

- 问题标题
- 文件名或任务编号
- 问题时间
- 问题描述
- 实际结果
- 期望结果
- 影响范围
- 提交角色

### 8.2 问题分级

- `P0`
  严重阻塞，系统不可用或核心链路中断
- `P1`
  关键结果错误，明显影响客户判断或交付可信度
- `P2`
  一般缺陷或体验问题，不影响主流程但影响使用感受
- `P3`
  优化建议或后续增强项

### 8.3 问题分类

- 抽取问题
- 结构召回问题
- 专题识别问题
- 汇总装配问题
- 页面展示问题
- 配置问题
- 数据样本问题
- 使用说明问题

### 8.4 问题流转

1. `O-运营` 负责建单与补全上下文
2. `M-管理` 负责定优先级与处理时限
3. `T-开发` 负责定位、修复、验证
4. `O-运营` 负责复测与反馈回传

### 8.5 问题状态

统一建议使用以下状态：

- `待受理`
- `已受理`
- `处理中`
- `待验证`
- `已关闭`
- `延期处理`
- `不纳入处理`

### 8.6 关闭标准

问题满足以下条件后才能关闭：

- 已有明确原因说明
- 已有修复动作或明确不处理结论
- 已完成验证
- 已向提出方反馈结果
- 如属共性问题，已判断是否需要纳入样本、评测或版本计划


## 9. 运营问题台账建议字段

建议至少包含以下字段：

- 编号
- 问题标题
- 来源
- 文件名
- 优先级
- 分类
- 当前状态
- 责任角色
- 责任人
- 发现时间
- 期望完成时间
- 实际完成时间
- 根因说明
- 处理结果
- 是否复发
- 是否纳入样本集
- 是否纳入版本计划


## 10. 复盘机制

以下问题建议必须复盘：

- 所有 `P0`
- 所有 `P1`
- 同类问题一周内重复出现两次及以上
- 质量门失败并影响发布判断的问题

复盘至少回答以下问题：

1. 问题为什么发生
2. 为什么没有提前发现
3. 以后通过什么机制避免再次发生
4. 是否需要补：
   - 样本
   - 测试
   - 质量门
   - 使用说明
   - 页面提示


## 11. 当前执行原则

现阶段项目执行时，统一遵循以下原则：

- 客户问题先闭环，再讨论表达
- 高优先级问题先止血，再做长期优化
- 任何修复都尽量沉淀到测试、样本或质量门
- 任何关闭都要有结论，不接受“没人继续跟”的自然沉没


## 12. 版本说明

- 当前版本：`V1`
- 生效范围：`test_getst` 项目
- 适用角色：`O-运营 / M-管理 / T-开发`
