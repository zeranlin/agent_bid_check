# Task-OG3-F 跨层冲突仲裁与最终准入补强任务单

## 基本信息

- 任务编号：`Task-OG3-F`
- 任务名称：跨层冲突仲裁与最终准入补强
- 任务类型：`治理架构实施 / 补强整改`
- 当前状态：`待下发`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

M 在首轮验收中发现，当前治理层虽然已经有：

- 风险家族
- 标题单源化
- 同层去重
- 成品页一致性测试

但 `OG3` 的关键目标“最终输出准入与真实文件门禁”还存在一个核心缺口：

当前风险家族的合并键仍带着 `target_layer`，因此同一风险家族如果跨层并存，例如：

- `formal`
- `pending`
- `excluded`

治理层不会做最终二选一或三选一，而是允许它们同时存在。

这意味着：

- 真实文件仍可能存在跨层冲突
- 最终准入控制还不是严格意义上的“唯一主归宿”

## 任务目标

1. 建立风险家族的跨层冲突仲裁机制
2. 让同一风险家族在最终结果中只保留一个最终归宿
3. 补齐真正的最终准入控制
4. 让真实文件门禁从“结果对齐”升级为“冲突不可回流”

## 具体整改要求

### 1. 建立跨层冲突仲裁

对于同一 `risk_family`，治理层必须能判断：

- 最终保留在哪一层
- 为什么保留在该层
- 为什么其他层级被覆盖或吸收

要求：

- 不允许同一家族同时出现在多个层级中
- 至少对 formal / pending / excluded 三层都适用

### 2. 定义跨层优先级规则

至少明确以下问题：

1. 何时 `formal` 优先
2. 何时 `pending` 优先
3. 何时 `excluded` 应覆盖前两者
4. 是否允许“正式风险 + 补充 pending”共存

要求：

- 必须写成可执行规则
- 不能只留在说明文档里

### 3. 强化最终准入控制

最终输出前，必须有一层显式检查：

- 是否存在同一风险家族跨层并存
- 是否存在已被覆盖的旧层级结果
- 是否存在不应进入成品页的残留对象

### 4. 补跨层冲突测试

至少增加以下场景：

- 同一风险家族同时进入 formal 和 pending，最终仅保留一个层级
- 同一风险家族同时进入 pending 和 excluded，最终仅保留一个层级
- 同一风险家族三层并存时，最终按规则只保留一个归宿

### 5. 强化真实文件门禁

真实文件门禁除现有一致性检查外，必须新增：

- 跨层冲突不存在
- 同一风险家族唯一归宿
- 被覆盖层级不会回流到 Web / Markdown / history

## 交付物要求

T 提交时必须附：

1. 跨层冲突仲裁规则说明
2. 最终准入控制说明
3. 新增跨层冲突测试清单
4. 真实文件门禁补强说明
5. 修复前后冲突示例对照

## M 验收标准

### 通过线

- 同一风险家族不再允许跨层并存
- 已建立明确的跨层优先级规则
- 成品页准入前已有冲突检查
- 真实文件门禁已覆盖跨层冲突回流

### 不通过情形

- 仍按 `(target_layer, family_key)` 各自保留
- 仍允许同一家族同时出现在多个层级
- 没有跨层冲突测试
