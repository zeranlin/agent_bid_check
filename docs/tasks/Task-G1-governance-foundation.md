# Task-G1 治理骨架搭建任务单

## 基本信息

- 任务编号：`Task-G1`
- 任务名称：治理骨架搭建
- 任务类型：`治理架构实施`
- 当前状态：`已通过`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

项目当前已经完成：

- 规则治理总体方案
- 规则注册表维护机制
- 规则注册表字段模板
- V2 规则治理架构实施清单

但这些内容目前仍主要停留在文档层，仓库中还没有形成正式的治理骨架。

当前最先需要落地的，不是继续堆规则，而是把“规则应该放哪里、以什么格式维护、状态如何流转、如何做基础校验”这些底座搭起来。

`Task-G1` 的目标，就是把治理方案从“文档定义”变成“仓库骨架”。

## 任务目标

1. 在仓库中建立正式的规则治理目录
2. 落地规则注册表模板文件
3. 定义规则状态流转的最小约束
4. 提供规则文件基础校验骨架
5. 让后续 `Task-G2 / Task-G3` 有统一的承接结构

## 具体整改要求

### 1. 建立规则治理目录

至少建立以下目录：

- `rules/registry/`
- `rules/templates/`
- `rules/schemas/`

如有必要，可补充：

- `rules/tools/`
- `rules/examples/`

要求：

- 目录职责清晰
- 不允许只是空目录而无说明
- 至少补一个说明文件或样板文件，方便后续接着用

### 2. 落地规则模板文件

基于现有治理文档，生成正式模板文件，例如：

- `rules/templates/rule_template.yaml`

模板至少覆盖：

- `rule_id`
- `rule_name`
- `status`
- `owner`
- `source`
- `classification`
- `trigger_conditions`
- `exclude_conditions`
- `downgrade_conditions`
- `output`
- `samples`
- `tests`
- `task_refs`
- `activation`
- `history`

要求：

- 字段命名与治理文档保持一致
- 模板内容可直接复制为新规则文件

### 3. 建立规则状态约束

至少支持以下状态：

- `draft`
- `in_progress`
- `review`
- `active`
- `rejected`
- `deprecated`

要求：

- 模板中已有状态字段
- 有最小状态说明
- 后续规则文件能统一套用

### 4. 建立规则基础校验骨架

至少提供一个轻量脚本或模块，支持校验：

- 是否存在必填字段
- 是否缺 `trigger_conditions`
- 是否缺 `exclude_conditions`
- 是否缺 `output.formal_title`
- 是否缺样本或测试引用字段

注意：

- 第一版不要求复杂平台化
- 但必须至少能对规则文件做结构完整性检查

### 5. 提供首个示例规则文件

至少创建 1 个示例规则文件，例如：

- `rules/registry/_example_rule.yaml`

要求：

- 用真实模板填充
- 字段示例完整
- 能作为后续规则迁移参考

## 必须补的测试

### 1. 规则模板可加载测试

断言：

- 模板文件可被正常读取
- 字段结构符合预期

### 2. 规则校验器测试

断言：

- 缺关键字段时校验失败
- 字段齐全时校验通过

### 3. 示例规则文件测试

断言：

- 示例规则文件可通过基础校验

## 交付物要求

T 提交时必须附：

1. 新增的目录清单
2. 新增的模板文件
3. 新增的校验脚本或模块
4. 示例规则文件
5. 测试文件与执行结果

## M 验收标准

### 通过线

- 仓库中已建立规则治理骨架目录
- 已存在可复用的规则模板文件
- 已有基础规则校验能力
- 已有至少 1 个示例规则文件
- 对应测试通过

### 不通过情形

- 只有文档，没有仓库骨架
- 有模板但字段不完整
- 无法校验规则文件完整性
- 没有测试支撑
