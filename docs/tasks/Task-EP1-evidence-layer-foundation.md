# Task-EP1 证据层骨架搭建任务单

## 基本信息

- 任务编号：`Task-EP1`
- 任务名称：证据层骨架搭建
- 任务类型：`治理架构实施 / 重构任务`
- 当前状态：`待下发`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

当前系统虽然已经具备：

1. 文档解析能力
2. 专题识别能力
3. 候选风险识别能力
4. 输出治理与风险准入能力

但真实文件持续回放证明，系统当前仍然大量默认：

`抽到一段原文 = 得到一个可直接裁决的风险证据`

这会导致：

1. 正文条款、模板条款、合同格式条款、附件条款、提醒项、留白项混在一起
2. 技术、商务、资格、评分、验收等业务类型经常在后续层级才被被动区分
3. 模板误报、提醒项误升、合同格式误报等问题不断复发

因此，本轮重构必须先在解析层和专题层之间补一层正式的 `证据层`，让文本先被建模为结构化证据对象，再进入专题和规则链路。

`Task-EP1` 是整轮重构的起点任务。

如果这一层骨架没有搭好，后续 `EP2 / EP3 / EP4`、问题层、发布层都会建立在不稳定对象之上。

## 任务目标

1. 建立正式的证据层模块边界
2. 定义统一的证据对象 schema
3. 打通“解析输出 -> 证据层 -> 专题识别层”的最小链路
4. 为后续来源分类、业务分类、条款角色、硬证据判定提供稳定承接底座

## 本任务范围

本任务只做“证据层骨架”和“最小链路打通”，不要求在本任务内一次性补齐全部业务分类规则。

本任务应完成：

1. 证据层目录与核心模块
2. 证据对象的最小必需字段
3. 主链路接线
4. 基础测试
5. 最小 replay 验证

本任务不要求完成：

1. 全量 `source_kind` 分类补齐
2. 全量 `business_domain` 分类补齐
3. 全量 `clause_role` / `hard_evidence` 规则补齐
4. 问题层归并
5. 发布层改造

这些分别由后续：

1. `Task-EP2`
2. `Task-EP3`
3. `Task-EP4`
4. `Task-PB1+`
5. `Task-PU1+`

继续承接。

## 具体整改要求

### 1. 建立证据层模块骨架

至少建立正式模块目录，例如：

- `app/pipelines/v2/evidence_layer/`

如项目现有目录更适合，也可采用等价结构，但必须满足：

1. 与解析层、专题层职责分离
2. 可独立测试
3. 可被后续 `EP2 / EP3 / EP4` 继续扩展

建议至少包含以下文件或等价模块：

- `schemas.py`
- `models.py`
- `classifier.py`
- `pipeline.py`
- `README.md`

### 2. 定义证据对象最小 schema

至少定义一个统一的 Evidence 对象，支持后续扩展。

最小字段至少包括：

| 字段 | 要求 |
| --- | --- |
| `evidence_id` | 唯一标识 |
| `excerpt` | 原文摘录 |
| `location` | 页码 / 章节 / 锚点 |
| `source_kind` | 来源类型占位字段 |
| `business_domain` | 业务域占位字段 |
| `clause_role` | 条款角色占位字段 |
| `evidence_strength` | 证据强弱占位字段 |
| `hard_evidence` | formal 支撑能力占位字段 |
| `topic_hints` | 后续专题可用提示 |
| `metadata` | 扩展字段 |

要求：

1. 本任务即使不能全部判准，也必须把字段结构先建出来
2. 后续分类任务必须复用该对象，不允许再次新起平行对象

### 3. 打通最小主链路

必须建立如下最小流程：

```text
document parse output
-> evidence layer
-> evidence objects
-> topic review input
```

要求：

1. 专题层后续不再直接以“原始切片列表”作为唯一输入
2. 至少有一条主链路能证明 evidence objects 已进入专题前置环节
3. 不允许证据层成为“仅新建目录但未接线”的空壳

### 4. 保留来源与业务分类的扩展挂点

虽然本任务不要求完成全量分类，但必须至少预留：

1. `source_kind` 分类挂点
2. `business_domain` 分类挂点
3. `clause_role` 分类挂点
4. `hard_evidence` 判定挂点

要求：

1. 后续 `EP2 / EP3 / EP4` 可以在此基础上增量接入
2. 不允许后续任务为了补分类再推翻本任务对象模型

### 5. 证据 trace 基础能力

至少补上最小证据追溯能力，使后续可以从候选风险回到：

1. `evidence_id`
2. 原文摘录
3. 原文位置

本任务不要求前端直接展示，但必须保证后续问题层和发布层能使用。

## 必须补的测试

至少补以下测试：

1. 证据对象构造测试
2. 证据层输出 schema 测试
3. 解析输出进入证据层测试
4. 证据层结果可进入专题输入测试
5. 证据 trace 基础链路测试

## 真实文件验证要求

至少选 1 份关键真实文件做最小 replay 验证。

推荐优先：

1. 柴油文件

验证目标不是“误报已全部下降”，而是证明：

1. 证据层已真正进入主链路
2. 系统已生成证据对象
3. 候选风险能追回证据对象
4. 本任务没有打坏当前 V2 输出

## 交付物要求

至少交付：

1. 证据层模块目录
2. 证据对象 schema 说明
3. 主链路接线说明
4. 新增测试清单
5. 最小 replay 结果说明

## M 验收标准

### 通过线

满足以下条件可判通过：

1. 已建立正式证据层模块，而不是临时工具函数
2. 已定义统一 Evidence 对象，字段足以承接后续分类任务
3. 主链路已打通，专题层前能看到证据对象
4. 已有最小 trace 能力，可从候选回到证据
5. 基础测试通过
6. 至少 1 份真实文件 replay 证明本任务未打坏现有输出

### 不通过情形

出现以下任一情况，本任务不能通过：

1. 只是新建目录，没有接入主链路
2. 仍然由专题层直接吃原始文本切片，证据对象未真正使用
3. 没有统一 schema，只是散落加几个字段
4. 证据层没有可扩展挂点，后续分类任务无法承接
5. 没有 replay 验证
