# Task-G6 候选规则分流与纳管台账搭建任务单

## 基本信息

- 任务编号：`Task-G6`
- 任务名称：候选规则分流与纳管台账搭建
- 任务类型：`治理架构实施`
- 当前状态：`待执行`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

当前项目正式规则库主要集中在 `R-001 ~ R-008`，规则数量和覆盖面都还偏薄。

外部系统提供了一批约 150 条审查点，这批内容对项目有补充价值，但它们并不是可以直接导入正式规则库的“成品规则”，而是混合了：

1. 可直接规则化的稳定禁限项
2. 依赖具体品类或场景的条件型规则
3. 需要专题能力承接的能力项
4. 应删除、暂缓或不纳入当前系统的条目

因此，第一步不是直接新增大量 `R-xxx`，而是先搭建候选规则分流与纳管台账，把候选池接住、分清、挂账、可追踪。

## 任务目标

1. 建立候选规则池的最小治理入口
2. 建立标准化候选规则分流台账
3. 将外部候选规则统一编号、统一来源、统一状态管理
4. 为后续 `Task-G7 / Task-G8` 提供稳定承接底座

## 具体整改要求

### 1. 建立候选规则治理目录

至少建立以下目录：

- `rules/candidates/`
- `rules/candidates/sources/`
- `rules/candidates/imports/`
- `rules/candidates/mappings/`
- `rules/candidates/snapshots/`

要求：

1. 候选池必须与 `rules/registry/` 正式规则库物理隔离
2. 每个目录必须有说明文件或最小样板文件
3. 本轮不允许候选条目直接进入正式运行链路

### 2. 建立候选规则导入样板

至少建立 1 份首版候选导入样板文件，承接外部审查点来源。

样板至少需要包含：

- 批次编号
- 来源文档名称
- 来源路径或来源说明
- 候选条目列表
- 每条候选条目的最小标识字段

建议字段：

- `candidate_id`
- `source_rule_text`
- `source_category`

### 3. 建立候选规则分流台账

至少建立 1 份首版候选规则分流台账，支持记录每条候选规则的治理判断。

台账至少包含以下字段：

- `candidate_id`
- `source_name`
- `source_rule_text`
- `source_category`
- `decision`
- `decision_reason`
- `target_rule_id`
- `target_layer`
- `profile_dependency`
- `negative_conditions`
- `samples_status`
- `tests_status`
- `task_id`
- `status`
- `snapshot_id`

### 4. 明确定义分流结论

本轮统一使用以下 4 类分流结论：

- `formal_rule`
- `conditional_rule`
- `capability_item`
- `drop`

要求：

1. 这 4 类必须在台账模板或说明文档中有清晰定义
2. 后续新候选条目必须复用这套口径，不允许临时发明新状态

### 5. 首轮接入少量样例条目

本任务不要求一次性录入全部 150 条，但必须：

1. 至少接入一小批样例候选条目
2. 能体现“正式规则 / 条件型规则 / 能力项 / 删除项”的分流方向
3. 能作为后续批量接入的模板

## 必须补的测试

### 1. 候选治理目录存在性测试

断言：

- 候选规则目录存在
- 说明文件存在

### 2. 候选导入样板可读取测试

断言：

- 首版导入样板可被正常读取
- 批次字段与候选字段存在

### 3. 候选分流台账字段完整性测试

断言：

- 首版台账可被正常读取
- 必填字段齐全
- 分流结论值在允许集合内

## 交付物要求

T 提交时必须附：

1. 候选规则目录清单
2. 候选导入样板文件
3. 候选分流台账文件
4. 台账字段说明
5. 测试文件与执行结果
6. 当前已接入的样例候选条目清单

## M 验收标准

### 通过线

- 候选池与正式规则库已明确隔离
- 候选规则导入样板已建立
- 分流台账字段完整且口径统一
- 分流结论定义清晰
- 对应测试通过

### 不通过情形

- 候选池与正式规则库混放
- 没有台账，只是随手写几条候选点
- 台账字段缺失严重
- 分流结论口径不统一
- 没有测试支撑
