# Task-OG1 输出治理骨架搭建任务单

## 基本信息

- 任务编号：`Task-OG1`
- 任务名称：输出治理骨架搭建
- 任务类型：`治理架构实施`
- 当前状态：`待下发`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

真实文件连续复核已经证明，当前系统的问题不再只是“规则有没有命中”，而是：

- 命中了，但重复报
- 命中了，但该降级没降级
- 命中了，但该删除没删除
- 规则修了，但最终成品页还是会回流旧口径

这说明现有 V2 虽然已经有：

- 解析层
- 专题识别层
- compare / 规则命中层

但还缺少一个真正负责“最终裁决”的输出治理中枢。

`Task-OG1` 的目标，是先把这个输出治理层的骨架搭起来，让后续 `Task-OG2 / Task-OG3` 有正式承接结构。

## 任务目标

1. 在现有 V2 链路中建立正式的输出治理模块边界
2. 定义风险身份、风险家族、治理结果对象的最小结构
3. 建立“规则命中结果 -> 输出治理结果 -> 最终输出”的明确链路
4. 为后续归并、分层、准入控制提供统一入口

## 具体整改要求

### 1. 建立输出治理目录或模块

至少建立一组正式承接模块，例如：

- `app/pipelines/v2/output_governance/`

如项目现有目录更适合，也可采用等价命名，但必须满足：

- 职责独立
- 不与 topic / rendering 混在一起
- 后续可继续扩展

### 2. 定义治理对象模型

至少定义以下结构化对象或等价数据结构：

- `RiskIdentity`
- `RiskFamily`
- `GovernedRisk`
- `GovernanceDecision`

要求至少覆盖以下信息：

- `rule_id`
- `risk_family`
- `source_topics`
- `evidence_anchors`
- `document_span`
- `proposed_title`
- `canonical_title`
- `target_layer`
- `governance_reason`

### 3. 打通最小链路

要求建立以下最小流程：

```text
compare / rule hits
-> output governance input
-> governance decision
-> governed result
```

第一版不要求所有规则都接入完整治理逻辑，但必须至少能：

- 接收 compare 结果
- 生成治理结果对象
- 输出统一结构供下游消费

### 4. 预留治理策略接口

至少预留以下策略入口：

- 身份识别
- 风险合并
- 风险分层
- 标题规范化
- 准入控制

第一版可以是骨架实现，但不能把后续逻辑继续散落回旧链路。

### 5. 建立最小说明文档

要求补一份开发侧说明，说明：

- 输出治理层放在哪里
- 输入是什么
- 输出是什么
- 与 compare、rendering 的边界是什么

## 必须补的测试

### 1. 治理对象可构造测试

断言：

- 治理输入可正常转换为治理对象
- 核心字段存在

### 2. 最小链路测试

断言：

- compare 命中结果能够进入治理层
- 治理层能产出统一结果对象

### 3. 边界测试

断言：

- topic 原始自由文本不会直接冒充治理结果对象

## 交付物要求

T 提交时必须附：

1. 新增模块目录清单
2. 治理对象模型说明
3. 最小链路说明
4. 新增测试清单
5. 一份结构示例输出

## M 验收标准

### 通过线

- 已建立独立输出治理骨架
- 已有明确治理对象模型
- compare 结果可进入治理层
- 下游已有统一治理结果可消费
- 对应测试通过

### 不通过情形

- 仍把治理逻辑散落在旧链路中
- 只有概念说明，没有正式模块
- 没有结构化治理对象
- 没有测试支撑
