# Task-OG3 最终输出准入与真实文件门禁任务单

## 基本信息

- 任务编号：`Task-OG3`
- 任务名称：最终输出准入与真实文件门禁
- 任务类型：`治理架构实施`
- 当前状态：`待下发`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

即使 compare 或治理层结果已经变对，如果：

- `review.md` 仍能吃旧 summary
- Web 卡片还能回流旧标题
- history 页面和最终报告不一致
- 真实文件没有纳入强门禁

那整改依然不算真正完成。

`Task-OG3` 的目标，是把“最终输出只能消费治理结果”做成硬约束，并把真实文件回放纳入正式门禁。

## 任务目标

1. 建立最终输出准入机制
2. 让 Markdown 报告、Web 页面、history 页面统一只消费治理结果
3. 建立真实文件回放门禁
4. 将“真实文件过线”纳入整改通过条件

## 具体整改要求

### 1. 建立输出准入控制

要求明确：

- 只有输出治理层产出的正式对象可进入最终业务输出
- topic summary、baseline summary、raw summary 不得直接进入成品页

至少覆盖：

- `final_output.json`
- `final_review.md`
- Web 风险卡片
- history 页面

### 2. 建立标题与分层一致性约束

要求：

- Web 标题与 Markdown 报告标题一致
- history 页面与最终报告分层一致
- `excluded_risks` 不得回流到正式风险或总结区
- `pending_review_items` 不得被自动提升为正式风险

### 3. 建立真实文件门禁

至少将以下对象纳入门禁：

- 当前柴油发电机组真实文件
- 后续已登记的关键真实文件案例

门禁断言至少包括：

- 正式风险数量收敛
- 重复报不回流
- 已确认误报不回流
- 已确认漏报能稳定命中

### 4. 建立整改通过新标准

后续凡涉及规则、输出、误报、漏报整改，默认通过标准至少包括：

1. 单元测试通过
2. 样本回归通过
3. 真实文件回放通过
4. Web / Markdown / history 三端一致

要求在代码、测试或文档中至少有一处正式体现，而不是只停留在口头约定。

### 5. 补充成品页测试

至少新增或补强：

- 最终报告装配测试
- Web 卡片一致性测试
- history 页面一致性测试
- 真实文件成品页回放测试

## 必须补的测试

### 1. 输出准入测试

断言：

- 非治理结果对象不能直接进入最终业务输出

### 2. 三端一致性测试

断言：

- Markdown 报告、Web 页面、history 页面标题与分层一致

### 3. 真实文件门禁测试

断言：

- 指定真实文件在整改后达到预期输出结果
- 已知回流问题不会再次出现

## 交付物要求

T 提交时必须附：

1. 输出准入链路说明
2. 三端一致性说明
3. 新增成品页测试清单
4. 真实文件门禁清单
5. 一份整改前后结果对照

## M 验收标准

### 通过线

- 最终输出已建立准入控制
- 三端输出已实现口径一致
- 真实文件已纳入正式门禁
- 后续整改通过条件已升级为“规则 + 测试 + 真实文件 + 成品页”四项闭环

### 不通过情形

- 最终报告仍能绕过治理层读取旧内容
- Web 和 Markdown 结果仍不一致
- 真实文件没有进入正式门禁
- 整改完成标准仍停留在“规则改了、测试过了”
