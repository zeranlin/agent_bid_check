# Task-G3 首批规则迁移任务单

## 基本信息

- 任务编号：`Task-G3`
- 任务名称：首批规则迁移
- 任务类型：`治理架构实施`
- 当前状态：`已通过`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

治理骨架和输出单源化建立后，下一步就要把现有高价值规则从“散落在代码、测试、任务单中的状态”迁移到正式规则注册表中。

`Task-G3` 的重点不是新增新规则，而是先把已经经过真实文件验证、已经有明确业务价值的核心规则迁移成正式治理资产。

## 本轮建议迁移范围

优先迁移以下规则：

1. `R-001` 补齐“拒绝进口 vs 外标引用”跨专题一致性风险识别
2. `R-002` 补齐“强制性标准条款未按评审规则加注★”风险识别
3. `R-003` 补齐“不得将项目验收方案作为评审因素”评分合规风险识别
4. `R-004` 补齐“不得将付款方式作为评审因素”评分合规风险识别
5. `R-005` 补齐“不得要求提供赠品、回扣或者与采购无关的其他商品、服务”评分合规风险识别
6. `R-006` 补齐“不得限定或者指定特定的专利、商标、品牌或者供应商”评分合规风险识别
7. `R-007` 补齐“不得要求中标人承担验收产生的检测费用”需求合规风险识别

## 任务目标

1. 将首批核心规则正式落入注册表
2. 为每条规则建立样本映射
3. 为每条规则建立测试映射
4. 为每条规则建立任务单映射
5. 让后续规则新增与旧规则补强都能按统一结构维护

## 具体整改要求

### 1. 建立首批规则注册文件

要求：

- 为上述规则分别建立独立规则文件
- 文件命名与规则编号对应
- 字段结构遵循统一模板

至少补齐：

- `rule_id`
- `rule_name`
- `status`
- `classification`
- `trigger_conditions`
- `exclude_conditions`
- `output`
- `samples`
- `tests`
- `task_refs`

### 2. 建立规则与样本映射

每条规则至少补齐：

- 1 个正样本
- 1 个负样本
- 如已存在真实文件案例，补 1 个 replay 样本

要求：

- 样本路径在规则文件中可追踪
- 不允许规则文件只写标题，不挂样本

### 3. 建立规则与测试映射

每条规则至少挂接：

- 1 条规则识别测试
- 1 条回放或真实文件测试

如暂时没有完整测试，也必须预留字段并说明后补计划。

### 4. 建立规则与任务单映射

每条规则必须能映射到：

- 主整改任务单
- 如有补强单，也要挂接

要求：

- 后续能从规则文件反查任务单
- 也能从任务单反查规则编号

### 5. 统一规则状态

本轮迁移时，建议按当前验收结果统一设置：

- 已通过并关单的规则，可标为 `active`
- 尚未稳定的规则，可标为 `review` 或 `in_progress`

要求：

- 状态必须和当前台账口径一致
- 不允许台账说已通过、规则文件还停在 `draft`

## 必须补的测试

### 1. 首批规则文件完整性测试

断言：

- 首批规则文件都存在
- 关键字段齐全

### 2. 样本映射完整性测试

断言：

- 每条首批规则至少有正样本和负样本路径

### 3. 任务映射完整性测试

断言：

- 每条首批规则都能找到对应任务单路径

## 交付物要求

T 提交时必须附：

1. 首批规则文件清单
2. 每条规则对应的样本清单
3. 每条规则对应的测试清单
4. 每条规则对应的任务单清单
5. 规则状态说明
6. 测试执行结果

## M 验收标准

### 通过线

- `R-001 ~ R-007` 已完成首批规则注册
- 每条规则都有样本、测试、任务单映射
- 规则状态与台账状态一致
- 对应完整性测试通过

### 不通过情形

- 只迁移部分规则且无说明
- 规则文件字段缺失严重
- 规则与样本/测试/任务单断链
- 规则状态与台账口径不一致
