# Task-G4 客户反馈闭环演练任务单

## 基本信息

- 任务编号：`Task-G4`
- 任务名称：客户反馈闭环演练
- 任务类型：`治理架构实施`
- 当前状态：`待验收`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

项目当前已经完成：

- `G-001` 治理骨架搭建
- `G-002` 输出单源化
- `G-003` 首批规则迁移

这意味着规则治理框架已经具备基础能力，但还没有验证一个最关键的问题：

> 当客户对真实文件审查结果提出反馈时，系统能否把“漏报 / 误报”稳定转化为规则调整、样本补齐、测试回归和最终输出修正？

项目的根目标不是“把治理文档写完整”，而是：

1. 精准找到文件里的风险点
2. 客户发现问题后，系统能快速调整
3. 调整结果能沉淀，不再重复犯同类错误

因此，`Task-G4` 的目标是用一次真实的“客户反馈 -> 系统修正 -> 回归验收”流程，把整条闭环跑通。

## 任务目标

1. 跑通一次完整的客户反馈闭环
2. 明确区分“漏报”与“误报”的处理路径
3. 验证当前规则治理机制已可承接真实反馈
4. 形成一套后续可复用的实际操作路径

## 本轮闭环范围

本轮必须至少覆盖以下两种场景中的各 1 条：

### 场景 A：漏报闭环

要求：

- 选择 1 条客户反馈的真实漏报问题
- 明确该问题属于：
  - 新规则缺失
  - 旧规则触发条件不足
  - 召回不足
  - 输出层漏打
- 将其进入正式整改流程

### 场景 B：误报闭环

要求：

- 选择 1 条客户反馈的真实误报问题
- 明确该问题属于：
  - 排除条件缺失
  - 降级条件缺失
  - 合同模板区 / 公告承接区边界未识别
  - 输出层回流
- 将其进入正式整改流程

## 具体整改要求

### 1. 建立客户反馈输入格式

至少定义一份统一的反馈记录格式，建议包含：

- 文件名称
- 反馈类型：漏报 / 误报
- 原文位置
- 客户意见
- M 的初步分析
- 初步归因层级
- 建议进入的任务类型

要求：

- 后续客户反馈不能继续散落在聊天记录中
- 必须有统一记录入口

### 2. 跑通漏报处理路径

至少完成以下动作：

1. 选定 1 条真实漏报反馈
2. M 做规则分析
3. 建规则或补规则任务
4. T 完成规则/召回/输出修正
5. 补样本和测试
6. 用真实文件回归
7. M 验收

要求：

- 必须明确最后到底补到了哪一层
- 必须说明该漏报后续如何避免再次出现

### 3. 跑通误报处理路径

至少完成以下动作：

1. 选定 1 条真实误报反馈
2. M 判断误报来源层级
3. 建整改任务
4. T 修正排除条件 / 降级条件 / 输出逻辑
5. 补负样本和测试
6. 用真实文件回归
7. M 验收

要求：

- 必须明确该误报是如何被抑制的
- 必须说明后续如何避免同类误报回流

### 4. 形成闭环记录

本轮结束后，必须能完整回答以下问题：

- 客户反馈进系统时，记录在哪里
- 是谁负责定性
- 是谁负责开发
- 修改了哪些规则 / 样本 / 测试 / 输出逻辑
- 如何验收
- 如何关单

### 5. 输出一份闭环矩阵

至少整理一份简表，列出：

- 反馈编号
- 反馈类型
- 文件
- 问题描述
- 归因层级
- 对应任务单
- 修复方式
- 验收结果

## 必须补的测试

### 1. 漏报回归测试

断言：

- 漏报问题修复后，真实文件或 replay 样本可以稳定命中

### 2. 误报阻断测试

断言：

- 误报问题修复后，不再进入正式风险
- 必要时应转入 pending 或 excluded

### 3. 闭环记录完整性测试或检查项

若暂不做自动化测试，至少提供人工检查清单，确认：

- 反馈已登记
- 任务已建立
- 样本已补
- 测试已补
- 验收已完成

## 交付物要求

T 提交时必须附：

1. 本轮选取的漏报案例
2. 本轮选取的误报案例
3. 对应的规则 / 逻辑修改清单
4. 新增样本清单
5. 新增测试清单
6. 真实文件回归结果目录
7. 一份客户反馈闭环矩阵

## M 验收标准

### 通过线

- 至少跑通 1 条漏报闭环和 1 条误报闭环
- 客户反馈已形成统一记录
- 修改结果已沉淀到规则 / 样本 / 测试 / 输出逻辑
- 真实文件结果能够体现修复效果
- 后续同类问题有明确的复用路径

### 不通过情形

- 只改代码，不记录客户反馈来源
- 只修当前文件，不沉淀为规则资产
- 只补规则，不补样本或测试
- 只说“以后按这个流程做”，但本轮没有真正跑通闭环
