## 任务背景

`Task-G9` 已把真实候选规则正式接入候选池，并给出 `first_priority_candidates`。下一步不能停留在“分流建议层”，而要把其中最稳定、最有业务价值的一小批条目，转成可下发的正式纳管入口。

本任务的重点不是一次性扩完全部候选，而是为 `R-009+` 和现有规则补强提供一个可执行的正式任务包。

## 任务目标

1. 从 `G9` 的 `first_priority_candidates` 中筛出首批建议正式纳管条目
2. 明确每条条目是映射到现有规则补强，还是建议新增 `R-009+`
3. 为建议新增规则输出核心草案
4. 为建议补强现有规则的条目明确补强归属
5. 输出一份可直接交给 `M` 下发后续 `R-009+` 的任务包

## 具体整改要求

### 1. 控制首批范围

1. 首批不追求数量
2. 首批优先纳入触发条件稳定、排除条件可定义、风险文案可统一的条目
3. 边界不清或与现有规则高度重叠的条目不强行推进

### 2. 输出首批正式纳管建议清单

每个建议条目至少要说明：

1. `candidate_id`
2. `source_rule_text`
3. 归属方式：`new_rule` 或 `strengthen_existing_rule`
4. 对应拟新增规则编号或现有规则编号
5. 当前纳管理由

### 3. 对拟新增规则输出草案

至少覆盖：

1. 规则名称
2. 规则目标
3. 触发条件草案
4. 排除条件草案
5. 标准风险文案草案
6. 整改建议草案

### 4. 对现有规则补强项输出归属说明

至少覆盖：

1. 补强目标规则
2. 需补强的触发边界
3. 与现有规则的边界关系
4. 为什么不单独新建规则

## 必须补的验证

1. 首批正式纳管建议清单存在
2. 每条建议项都有明确归属
3. 拟新增规则项的核心字段草案齐全
4. 映射到现有规则补强的条目有明确归属关系

## 交付物要求

1. 首批正式纳管建议清单
2. 每条建议纳管项的归属说明
3. 拟新增规则草案
4. 现有规则补强归属清单
5. 风险与遗留问题说明
