# Task-G2 输出单源化任务单

## 基本信息

- 任务编号：`Task-G2`
- 任务名称：输出单源化
- 任务类型：`治理架构实施`
- 当前状态：`已通过`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

当前项目已经多次暴露出同一类问题：

- compare 分层是对的
- 但最终 `review.md`、Web 页面、历史页仍可能回流旧口径

`W-006 / W-006-F` 已经在真实文件上证明，这不是单条规则问题，而是最终输出链路没有完全做到“单一事实源”。

因此，`Task-G2` 的目标不是再补新规则，而是确保所有最终业务可见输出只消费统一的最终归一结果。

## 任务目标

1. 让最终 `review.md` 只消费最终分层结果
2. 让 Web 卡片和历史页与最终报告一致
3. 禁止 topic 原始 summary、baseline 旧结论直接穿透到最终业务输出
4. 为最终业务成品页补齐自动化测试

## 具体整改要求

### 1. 固化最终报告装配逻辑

最终 `review.md` 必须只基于：

- `comparison.clusters`
- `comparison.metadata.pending_review_items`
- `comparison.metadata.excluded_risks` 的过滤结果

要求：

- 正式风险区只来自正式风险主清单
- 待补证复核区只来自 pending 项
- 已剔除误报不得回流到最终报告结论区

### 2. 清理旧摘要穿透问题

检查并清理以下路径：

- topic summary
- baseline summary
- raw summary
- 自由拼接的旧结论文本

要求：

- 这些内容可以作为内部 trace
- 但不能直接进入最终业务报告结论区

### 3. Web 展示与最终报告统一

至少确保以下对齐：

- Web 风险卡片标题
- Web 历史页风险标题
- 下载的 `review.md`

要求：

- 三者基于同一份最终结果
- 不允许 Web 看起来一套、Markdown 报告另一套

### 4. 总结区逻辑收敛

最终报告中的：

- `高风险问题`
- `中风险问题`
- `需人工复核事项`

必须分别对应：

- 正式风险中的高风险
- 正式风险中的中风险
- pending 项

不得：

- 把 excluded 项混入总结区
- 把 pending 项改写成正式风险
- 把 topic 自由摘要直接拼进总结区

## 必须补的测试

### 1. 最终报告 markdown 测试

断言：

- 总结区只覆盖最终分层结果
- excluded 内容不进入最终总结区
- pending 内容以复核项形式出现

### 2. Web 标题一致性测试

断言：

- Web 风险卡片标题与最终报告标题一致
- 不存在 Web 页面回流旧口径、报告已修正但页面未修正的问题

### 3. 真实文件成品页测试

固定真实文件回放，断言：

- 真实文件最终报告与 compare 分层一致
- 已知回流误报不再重新出现

## 交付物要求

T 提交时必须附：

1. 改动的输出装配文件清单
2. Web / report / history 的统一说明
3. 新增的最终成品页测试
4. 一份真实文件输出目录
5. 一份“整改前后最终报告对照表”

## M 验收标准

### 通过线

- 最终 `review.md` 已实现单源消费
- Web 与 `review.md` 口径一致
- excluded 项不再回流到最终业务输出
- 已有最终成品页级别测试

### 不通过情形

- compare 正确但最终报告仍错误
- Web 与 Markdown 报告不一致
- 旧摘要仍能绕过分层规则直接输出
- 没有成品页级测试
