# Task-G5 客户反馈正式接入运行任务单

## 基本信息

- 任务编号：`Task-G5`
- 任务名称：客户反馈正式接入运行
- 任务类型：`治理架构实施`
- 当前状态：`已通过`
- 下发对象：`T`
- 监督角色：`M`

## 任务背景

项目当前已经完成：

- `G-001` 治理骨架搭建
- `G-002` 输出单源化
- `G-003` 首批规则迁移
- `G-004` 客户反馈闭环演练

其中 `G-004` 已经证明：

- 客户反馈可以被统一记录
- 漏报和误报都可以进入闭环
- 规则 / 样本 / 测试 / 真实回归 / 验收可以打通

下一步不应再停留在“演练模式”，而应正式进入“运行模式”。

`Task-G5` 的目标，就是把下一条真实客户反馈作为正式运行单接入，而不是作为一次性演练案例处理。

## 任务目标

1. 将下一条真实客户反馈正式纳入标准流程
2. 让客户反馈处理从“演练”切换为“常态运行”
3. 建立持续反馈台账或统一入口
4. 明确后续每条真实反馈都如何进入系统

## 本轮范围

本轮至少要完成以下 3 件事：

### 1. 接入 1 条真实客户反馈

要求：

- 必须是真实的新反馈，不再复用 `G-004` 演练案例
- 必须完成正式登记
- 必须判断是漏报还是误报

### 2. 建立常态化反馈台账

要求：

- 不再只靠单次矩阵文件
- 要有一份后续可持续追加的总表

建议至少包含：

- 反馈编号
- 反馈日期
- 文件
- 反馈类型
- 问题描述
- 归因层级
- 对应规则
- 对应任务单
- 当前状态
- 验收结果

### 3. 跑通 1 条正式运行单

要求：

- 从反馈登记开始
- 到任务创建
- 到修复
- 到回归
- 到验收

必须完整走完，不接受只登记不修复，或只修复不登记。

## 具体整改要求

### 1. 固化正式接入入口

要求：

- 明确客户反馈正式记录放在哪里
- 明确命名规范
- 明确编号规则

建议：

- `feedback/records/FB-YYYYMMDD-xxx-*.yaml`

### 2. 建立反馈总台账

建议文件：

- `feedback/records/feedback-ledger.md`

要求：

- 后续每条真实反馈都必须登记
- 要能快速看出哪些已处理、哪些待处理、哪些已验收

### 3. 明确反馈到任务的映射规则

至少要区分：

- 新规则接入
- 旧规则补强
- 误报抑制
- 输出层修正

要求：

- 每条反馈都要明确进入哪类任务
- 不允许只写“待处理”，不说明进入路径

### 4. 完成一条正式反馈闭环

至少完成：

1. 真实反馈登记
2. M 初步分析
3. 任务单建立
4. T 开发修复
5. 样本补齐
6. 测试补齐
7. 真实文件回归
8. M 验收

### 5. 输出正式运行说明

至少整理一份简明说明，回答：

- 后续客户反馈怎么登记
- 谁负责分析
- 谁负责开发
- 谁负责验收
- 台账怎么更新

## 必须补的测试 / 检查项

### 1. 反馈记录格式检查

断言：

- 新真实反馈记录符合统一模板

### 2. 反馈总台账检查

断言：

- 新反馈已进入总台账
- 状态字段完整

### 3. 正式运行单回归检查

断言：

- 本轮正式运行的那条反馈，已形成对应样本、测试与回归结果

## 交付物要求

T 提交时必须附：

1. 本轮真实客户反馈记录
2. 反馈总台账
3. 对应任务单
4. 修复文件清单
5. 新增样本清单
6. 新增测试清单
7. 真实回归结果目录
8. 一份“正式运行说明”

## M 验收标准

### 通过线

- 已有常态化反馈台账
- 已接入 1 条真实客户反馈
- 该反馈已跑完整条正式闭环
- 结果已沉淀到规则 / 样本 / 测试 / 回归 / 台账
- 后续流程已经可持续复用

### 不通过情形

- 仍停留在演练模式
- 没有真实新反馈接入
- 只有登记，没有修复闭环
- 只有修复，没有台账和长期入口
