# V2 规则与审查架构调整方案

## 1. 背景

随着项目进入真实文件持续复核阶段，当前系统暴露出的主要问题，已经不再只是“某一条规则没识别到”，而是逐步演变为以下几类系统性问题：

1. 新规则补进去了，但最终报告仍复用旧口径
2. compare 分层已经正确，但 `review.md` / Web 展示又把已剔除误报重新写回
3. 同一风险在多个专题中被重复输出，缺少统一归并中心
4. 人工复核已经确认的误报、漏报，没有沉淀成可追踪、可回归、可验收的系统资产
5. 新规则、新误报、新真实文件复核结果之间，缺少统一的任务化管理机制

这说明当前架构虽然已经具备“三层审查”的基础能力，但还没有形成“规则中心化、风险归一化、输出单源化、整改闭环化”的成熟治理结构。

因此，本方案的目标不是推翻现有 V2，而是在现有三层能力之上，补齐面向持续规则演进与误报治理的管理型架构。

## 2. 当前主要架构问题

### 2.1 规则定义分散

当前规则可能分散在：

- prompt
- compare 逻辑
- 文案拼装逻辑
- 测试样本
- 人工任务单

问题在于：

- 同一条业务规则没有单一事实源
- 规则修改后，容易出现“识别逻辑改了，但输出文案没改”“compare 改了，但 Web 没改”

### 2.2 风险归并与分层不独立

当前“识别到了什么”和“最后要怎么报给客户”之间，还没有完全拆层。

直接后果：

- 同一组证据会在多个专题中重复报出
- 证据不足类问题容易被直接写成正式风险
- 已确认误报虽然在 compare 层被降级，但最终报告层又回流

### 2.3 最终输出没有做到单源消费

当前 `review.md`、Web 页面、历史页，并不总是严格只消费“最终归一后的风险结果”，仍可能夹带：

- topic summary
- baseline 旧结论
- 自由文本摘要

结果就是：

- compare 对了，最终页面仍可能错
- 风险分层对了，业务看到的报告仍可能混乱

### 2.4 误报/漏报治理缺少产品化闭环

目前 M 与 T 已经形成了事实上的协作机制：

- M 发现问题
- 出任务单
- T 修复
- M 验收

但这个机制还主要靠人工推动，没有固化为项目架构的一部分。

## 3. 调整目标

本次架构调整的总体目标有 4 个：

1. 让业务规则成为独立、可维护、可追踪的系统资产
2. 让风险归并、风险分层成为独立中间层，而不是散落在各处
3. 让所有最终输出只消费同一份最终结构化结果
4. 让“新规则接入 + 误报整改 + 验收关单”成为标准流程

## 4. 调整后的总体分层

建议将现有体系调整为以下 6 层：

### 第 1 层：文档解析与证据抽取层

职责：

- 文档解析
- 章节切分
- 结构映射
- 证据片段抽取
- 基础信号识别

产出物：

- `document_map`
- `structure_map`
- `evidence_map`
- 原始证据片段

这一层不直接决定“是否正式报风险”，只负责把证据尽量完整、稳定地抽出来。

### 第 2 层：专题识别层

职责：

- 按专题做候选风险识别
- 为每个专题产出候选风险、证据、缺失点、结构化信号

产出物：

- `topic_reviews/*.json`
- baseline 候选风险
- 专题 summary

这一层允许“多报一些候选项”，但不负责最终业务口径。

### 第 3 层：规则中心层

职责：

- 管理所有业务规则
- 每条规则形成统一注册信息
- 维护规则编号、触发条件、排除条件、标准文案、测试样本映射

每条规则建议至少包含：

- `rule_id`
- `rule_name`
- `rule_type`
- `trigger_signals`
- `blocking_conditions`
- `downgrade_conditions`
- `formal_title`
- `pending_title`
- `fix_suggestion`
- `sample_refs`
- `test_refs`

这一层是后续“客户新增规则接入”的核心入口。

### 第 4 层：风险归一与分层层

职责：

- 同类风险合并
- 统一标题
- 统一文案
- 风险分层
- 已知误报抑制
- 模板区 / 公告承接区 / 证据不足区降级

这一层应输出明确三分层：

1. 正式风险
2. 待补证复核
3. 已剔除误报

这一层是当前最需要补强的核心。

### 第 5 层：最终输出装配层

职责：

- 生成 `review.md`
- 生成 Web 展示卡片
- 生成历史页数据
- 生成摘要视图

关键原则：

- 只能消费第 4 层的最终结果
- 不允许直接拼接 topic 原始 summary 作为最终结论
- 不允许 `excluded_risks` 回流到业务输出

### 第 6 层：监督与治理层

职责：

- M/T 协作
- 任务单管理
- 验收单管理
- 台账管理
- 真实文件回归
- 样本沉淀

这一层不属于“模型推理层”，但属于项目长期稳定演进的必要层。

## 5. 调整后的核心原则

### 5.1 规则单一事实源

以后每条规则都应能在一个地方查到：

- 它是什么
- 为什么存在
- 触发条件是什么
- 不该触发的条件是什么
- 最终标准文案是什么
- 对应哪些任务和测试

### 5.2 最终报告单一事实源

以后所有业务可见输出都应只来源于“最终归一与分层层”的结构化结果。

不允许：

- topic 直接写结论到最终报告
- baseline 旧标题直接穿透
- 人工补摘要绕过分层结果

### 5.3 误报也是正式资产

误报不能只停留在聊天记录里，必须沉淀为：

- 一条治理结论
- 一份任务单
- 一组负样本
- 一条回归测试
- 一次验收记录

### 5.4 新规则接入必须任务化

客户新增规则后，不应直接临时改代码，而应走标准流程：

1. M 解析规则
2. 生成规则接入任务单
3. T 开发
4. M 验收
5. 关单并沉淀到规则中心、样本、测试、台账

## 6. 建议的目录拆分

建议新增或强化以下目录分工：

### 6.1 规则目录

建议：

- `rules/registry/`
- `rules/schemas/`
- `rules/mappings/`

用途：

- 存放规则注册信息
- 存放规则结构定义
- 存放规则到风险标题、测试样本、任务单的映射

### 6.2 风险归一目录

建议：

- `app/pipelines/v2/risk_resolution/`

用途：

- 风险合并
- 分层
- 去重
- 模板区剔除
- 误报抑制

### 6.3 输出装配目录

建议：

- `app/pipelines/v2/rendering/`

用途：

- 专门负责最终报告、Web 卡片、历史页渲染
- 明确禁止直接消费 topic 原始自由文本

### 6.4 治理文档目录

建议：

- `docs/governance/`
- `docs/tasks/`
- `docs/acceptance/`

用途：

- 治理方案
- 任务单
- 验收单

### 6.5 样本与回归目录

建议：

- `data/examples/rules/`
- `data/examples/real_cases/`
- `tests/rules/`
- `tests/reports/`

用途：

- 按规则维度沉淀样本
- 按真实文件沉淀回归
- 按最终报告沉淀渲染测试

## 7. 新规则接入的标准流程

建议以后统一采用以下流程：

### 步骤 1：业务输入新规则

输入来源：

- 客户新增规则
- 法规更新
- 真实文件新发现

### 步骤 2：M 做规则分析

M 输出：

- 规则含义拆解
- 风险标题建议
- 正样本/负样本建议
- 应命中的层级
- 是否属于新规则、旧规则补强、还是误报抑制

### 步骤 3：生成任务单

任务编号建议继续采用：

- `R-xxx`：规则接入整改
- `W-xxx`：真实文件 / 输出 / Web / 分层整改
- `E-xxx`：样本 / 评测 / 验收基线整改

### 步骤 4：T 开发实现

T 必须同步提交：

- 规则注册
- 代码实现
- 样本
- 测试
- 真实文件回归结果

### 步骤 5：M 验收

M 验收至少检查：

- 规则是否命中
- 负样本是否不过报
- 真实文件是否生效
- 最终报告是否对齐

### 步骤 6：关单沉淀

关单时同步更新：

- 台账
- 规则注册表
- 样本
- 回归测试
- 治理文档

## 8. M / T 的职责边界

### M 的职责

- 规则分析
- 问题定性
- 任务拆解
- 验收
- 关单
- 治理文档维护

### T 的职责

- 技术实现
- 样本补齐
- 测试补齐
- 真实文件复跑
- 交付变更说明

### 协作原则

- M 不直接把“主观讨论结论”当成通过条件，必须落到任务单与验收标准
- T 不以“compare 过了”替代“最终业务报告过了”
- 所有争议都回到“真实文件结果 + 测试 + 任务单要求”三件套

## 9. 近期最值得先做的架构动作

如果按价值和优先级排序，我建议先做 3 件事：

### P0：做“最终输出单源化”

目标：

- `review.md`
- Web 卡片
- history 页面

全部只消费最终归一结果，不再吃 topic 原始自由文本。

这是当前最值钱的，因为它能直接消灭“compare 对了，页面还错”的问题。

### P1：做“规则注册中心”

目标：

- 所有新规则统一注册
- 每条规则能映射到样本、测试、任务单

这是后续持续新增规则的基础。

### P2：做“误报治理中心”

目标：

- 已确认误报形成抑制规则
- 合同模板区、公告承接区、证据不足区形成统一降级框架

这是后续把真实文件结果真正做稳的关键。

## 10. 管理层汇报口径

如果对领导或技术负责人汇报，可以用一句话概括本次调整：

> 项目将从“专题驱动的风险识别系统”，升级为“规则中心驱动、风险归一控制、输出单源消费、监督闭环治理”的持续演进型审查平台。

再展开就是 4 个关键词：

1. 规则中心化
2. 风险归一化
3. 输出单源化
4. 整改闭环化

## 11. 结论

当前项目不需要推倒重做，但确实需要从“能识别风险”升级到“能稳定维护规则、抑制误报、统一输出、可持续验收”的下一阶段架构。

如果不做这次调整，后面每新增一条规则，都可能继续出现：

- 规则命中了，但输出没改
- compare 对了，但页面错了
- 误报修过了，但别的层又回流
- 真实文件验收结论无法长期稳定

因此，这次架构调整是必要的，而且已经具备明确的实施价值。
