# 项目管理分层方案

## 1. 目标

本方案用于解决两个问题：

1. 项目后续会持续新增“业务功能”，不能和“规则治理机制”混在一起。
2. 项目总目标是持续提升“风险点精准识别能力”，因此必须把规则治理、质量回归、产品功能分层管理。

一句话定位：

`业务模块负责对外提供能力，治理模块负责让能力持续变强，质量模块负责保证结果稳定可信。`

## 2. 顶层分层

建议项目按 4 个一级层管理：

### A. Product Layer

面向客户的业务功能层。  
回答：`系统对外提供什么能力`

包含：

- 招标文件审查
- review-plus Web
- 风险卡片展示
- 导出与历史记录
- 后续新增业务功能

### B. Governance Layer

规则治理与任务治理层。  
回答：`新规则怎么进入系统，怎么开发，怎么验收，怎么上线`

包含：

- 规则登记
- 规则编号
- 工单生成
- 台账跟踪
- 验收关单
- 规则入库

### C. Quality Layer

质量与回归层。  
回答：`如何证明系统真的变强了`

包含：

- 样本集
- 真实文件基线
- 回归测试
- 覆盖率统计
- 漏检归因

### D. Reporting Layer

汇报与运营层。  
回答：`当前能力成熟度如何，哪些已覆盖，哪些还缺`

包含：

- 架构说明
- 成熟度评估
- 风险覆盖率报表
- 管理汇报材料

## 3. 管理原则

所有新增事项都先做两步判断：

### 第一步：判断是“业务功能”还是“治理能力”

- 新页面、新交互、新专题、新导出能力：归 `Product`
- 新规则、新任务单、新验收机制、新回归基线：归 `Governance / Quality`

### 第二步：判断归属哪个业务域

例如：

- `review`
- `web`
- `search`
- `analytics`
- `report`

这样以后即使新增新业务，也不会冲掉规则治理体系。

## 4. 模块职责

### 4.1 Product Layer

职责：

- 承载客户实际使用的功能
- 组织前后端能力与业务流程
- 输出最终用户可见结果

典型对象：

- 文档上传与审查
- 风险点汇总
- Web 历史页
- 结果导出

### 4.2 Governance Layer

职责：

- 接收新规则
- 编号归档
- 生成任务单
- 跟踪状态
- 验收关单
- 规则入库

典型对象：

- `R-xxx`
- `W-xxx`
- `E-xxx`
- 总台账
- 规则库

### 4.3 Quality Layer

职责：

- 管理样本和测试
- 管理真实文件基线
- 监控回归和回退
- 做漏检归因

典型对象：

- 样本集
- regression
- real file baseline
- 命中矩阵
- 覆盖率统计

### 4.4 Reporting Layer

职责：

- 面向管理层和运营层输出可理解材料
- 做成熟度、覆盖率、风险趋势说明

典型对象：

- 架构图
- 路线图
- 汇报版文档
- 仪表盘口径说明

## 5. 角色分工

### 客户 / 业务

负责：

- 提新规则
- 提真实文件
- 提漏检反馈
- 提优先级

### M

负责：

- 规则分析
- 编号归档
- 任务单生成
- 台账维护
- 验收评分
- 规则入库

### T

负责：

- 规则实现
- 样本补充
- 测试补充
- 回归运行
- 结果回传

角色关系固定为：

`客户提问题，M管治理，T管实现。`

## 6. 编号体系

建议正式固定：

- `R-xxx`：规则识别整改
- `E-xxx`：样本 / 评测整改
- `W-xxx`：Web / 展示 / 链路整改
- `P-xxx`：产品功能任务
- `Q-xxx`：质量治理任务
- `O-xxx`：运营 / 汇报任务

说明：

- 编号是“治理视角”
- 业务域是“产品视角”

例如：

- `R-007`：规则整改，属于 `governance/rules`
- `W-003`：展示链路整改，属于 `governance/web`
- 未来新增“法规检索页”可定义为 `P-001`，属于 `product/search`

## 7. 生命周期

建议所有事项统一走 8 个阶段：

1. 提出
2. 分析
3. 归类
4. 编号
5. 开发
6. 验收
7. 入库 / 上线
8. 回归追踪

其中：

- Product 事项重点看“是否可用”
- Governance 事项重点看“是否可管”
- Quality 事项重点看“是否可验证”

## 9. 核心文档清单

建议至少维护以下 6 份核心文档：

1. `规则总目录`
   - 所有已生效规则
   - 编号、规则码、专题、状态、样本、测试映射
2. `规则接入流程`
   - 客户提规则后如何进入系统
3. `任务单模板`
   - 统一 `R/E/W/P/Q/O` 任务格式
4. `验收模板`
   - 统一通过线、不过线、打分规则
5. `真实文件基线说明`
   - 如何定义应命中 / 不应命中
6. `项目路线图`
   - P0 / P1 / P2 的演进方向

## 10. 新增业务功能时的归类规则

以后如果再新增一个功能，按下面规则判断：

### 属于 Product

如果它回答的是：

- 用户能多做什么
- 页面多了什么能力
- 系统多了什么输出

例如：

- 新页面
- 新导出
- 新专题
- 新统计功能

### 属于 Governance / Quality

如果它回答的是：

- 新规则怎么进入系统
- 如何验证系统变强
- 如何追踪回退
- 如何做工单流转

例如：

- 新规则工单
- 新回归基线
- 新验收逻辑
- 新命中矩阵

## 11. 当前建议的组织结论

当前项目应明确为三套并行系统：

### 1. 审查产品系统

面向客户，持续加业务功能

### 2. 规则治理系统

面向内部，持续吸收新规则并驱动开发闭环

### 3. 质量验证系统

面向稳定性，持续证明识别能力没有回退

这三者彼此关联，但管理上必须独立。

## 12. 当前最优先落地项

建议先做以下 5 件事：

1. 固定顶层分层定义
2. 固定编号体系
3. 固定目录边界
4. 固定任务单与验收模板
5. 建第一版规则总目录

## 13. 结论

建议将项目管理模型正式定义为：

`产品层 + 治理层 + 质量层 + 汇报层`

其中：

- 产品层负责“做功能”
- 治理层负责“管规则”
- 质量层负责“保准确”
- 汇报层负责“讲清楚”

这样以后无论新增多少业务功能，还是新增多少规则，都不会和“规则管理机制”混淆。
