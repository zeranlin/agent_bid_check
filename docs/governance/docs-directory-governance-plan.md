# docs/ 目录治理方案

## 1. 文档目标

本方案用于把当前 `docs/` 目录从“顶层持续堆文档”治理为“按职责分层、可持续扩展”的目录体系。

目标不是立即大搬家，而是先解决 4 个问题：

1. 新文档应该放哪里
2. 当前已有文档未来应归到哪里
3. 哪些文档属于长期资产，哪些属于过程材料
4. 迁移时如何尽量不打断当前使用链路

## 2. 目标目录结构

```text
docs/
├── README.md
├── architecture/
├── governance/
├── trackers/
├── tasks/
├── reviews/
├── reports/
├── templates/
└── archive/
```

### 2.1 `docs/architecture/`

职责：

- 存放系统架构、分层链路、模块关系、输入输出与产物关系说明

长期性：

- 长期有效

典型文档：

- V1 / V2 架构方案
- 分层处理设计
- 结构-专题-汇总链路说明

### 2.2 `docs/governance/`

职责：

- 存放规则治理、角色协作、接入流程、规范、目录治理等“机制类文档”

长期性：

- 长期有效

典型文档：

- 规则接入流程
- 规则注册表规范
- 角色协作机制
- 文档治理方案

### 2.3 `docs/trackers/`

职责：

- 存放长期维护的总台账、backlog、路线图、推进计划

长期性：

- 长期维护，但内容高频变化

典型文档：

- 总台账
- backlog
- 路线图
- 分阶段实施计划

### 2.4 `docs/tasks/`

职责：

- 存放正式任务单、整改单、补强单、治理实施单

长期性：

- 中长期保留，但以单次任务为颗粒度

典型文档：

- `R-*`
- `W-*`
- `G-*`
- `DOC-*`

### 2.5 `docs/reviews/`

职责：

- 存放真实文件逐条复核、人工验收笔记、单文件对账记录

长期性：

- 中长期保留，按文件/批次组织

### 2.6 `docs/reports/`

职责：

- 存放面向汇报和总结的文档，包括架构汇报版、评测说明、阶段汇报

长期性：

- 中长期保留

### 2.7 `docs/templates/`

职责：

- 存放可重复套用的文档模板

长期性：

- 长期有效

### 2.8 `docs/archive/`

职责：

- 存放失效方案、旧版本、迁移后历史留存文档

长期性：

- 历史归档

## 3. 文档类型落位规范

### 3.1 新架构图文档放哪里

- 放 `docs/architecture/`
- 如果是“架构汇报版”，放 `docs/reports/`

### 3.2 新规则治理文档放哪里

- 放 `docs/governance/`

### 3.3 新任务单放哪里

- 放 `docs/tasks/`

### 3.4 新验收记录或复核记录放哪里

- 单文件复核、逐条核对：`docs/reviews/`
- 长期台账式验收汇总：`docs/trackers/`

### 3.5 新 backlog 或路线图放哪里

- 放 `docs/trackers/`

### 3.6 历史失效方案放哪里

- 放 `docs/archive/`

## 4. 顶层保留策略

治理后顶层 `docs/` 建议只保留：

1. `README.md`
2. 少量仍未迁移、但被大量引用的存量文档
3. 正在过渡期的入口文档

原则：

- 顶层不再新增普通说明文档
- 新文档优先进入二级目录

## 5. 命名建议

建议统一命名风格：

- 任务单：`R-001.md`、`W-002.md`、`DOC-001.md`
- 复核记录：`<file-id>-review-notes-YYYY-MM-DD.md`
- 台账：`<scope>-tracker.md`
- backlog：`<scope>-backlog.md`
- 方案：`<scope>-plan.md`
- 汇报：`<scope>-report.md`

## 6. 迁移原则

### 第一步：补结构和索引，不大规模移动

本轮只做：

1. 建目录
2. 建入口
3. 建映射表
4. 建迁移清单

不做：

- 大规模 rename
- 大规模路径迁移
- 全量链接修复

### 第二步：按批次迁移老文档并修复链接

迁移顺序建议：

1. 先迁顶层 backlog / tracker / review 类文档
2. 再迁顶层 architecture / report 类文档
3. 最后处理旧版/失效文档归档

## 7. 本轮执行边界

本轮以“治理设计完成”为通过线。

已实施：

1. 目录骨架建立
2. 总入口建立
3. 现状分类映射建立
4. 迁移清单建立

暂不实施：

1. 批量移动存量文档
2. 大规模修复交叉链接
3. 归档旧文档正文改写
