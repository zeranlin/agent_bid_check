# V2 规则治理架构实施清单

## 1. 目的

本清单用于把现有治理方案从“设计文档”推进到“实际落地执行”。

当前项目已经形成以下方案基础：

- `V2 规则与审查架构调整方案`
- `规则注册表维护机制`
- `规则注册表字段模板`

下一步不再停留在讨论层，而是进入实施阶段。

本清单的目标是明确：

1. 先做什么
2. 后做什么
3. 每一步的产出物是什么
4. 每一步怎样算完成

## 2. 实施总原则

### 原则 1：先做治理骨架，再做规则迁移

不要先大规模迁移规则，再补基础设施。  
应先把目录、模板、状态、校验、输出单源化骨架搭起来。

### 原则 2：先稳住最终输出，再扩规则平台

当前最值钱的问题，是避免“compare 对了，最终报告还错”。  
所以最终输出单源化优先级高于复杂后台化。

### 原则 3：先做 MVP，再做自动化增强

第一版先接受：

- 人工创建规则内容
- 半自动维护流程
- 文件级注册表

先不追求：

- 规则后台 UI
- 自动审批流
- 完整平台化配置中心

## 3. 总体实施路线

建议分三阶段推进：

### P0：治理底座落地

目标：

- 让规则有注册位置
- 让输出有单一来源
- 让任务、台账、验收开始成体系

### P1：规则迁移与闭环成型

目标：

- 把现有高价值规则迁入注册表
- 建立规则到样本、测试、任务单的映射
- 初步形成“新增规则 -> 开发 -> 验收 -> 生效”闭环

### P2：自动化辅助与平台化增强

目标：

- 自动编号
- 自动校验
- 自动生成模板
- 自动同步台账
- 逐步向规则治理工具化发展

## 4. P0 实施清单

P0 是当前最应该先做的部分。

### P0-1 建立规则注册表目录

目标：

- 在仓库中正式建立规则注册区域

建议目录：

- `rules/registry/`
- `rules/templates/`
- `rules/schemas/`

产出物：

- 规则目录骨架
- 首份示例规则文件

验收线：

- 仓库中已存在规则目录
- 已有至少 1 个可参考的规则样例文件

### P0-2 固化规则文件模板

目标：

- 把规则字段模板真正落成可复用文件模板

建议产出：

- `rules/templates/rule_template.yaml`

内容应覆盖：

- 规则编号
- 规则名称
- 状态
- 触发条件
- 排除条件
- 降级条件
- 标准风险文案
- 整改建议
- 样本引用
- 测试引用
- 任务单引用

验收线：

- 模板文件存在
- 新规则可以直接复制模板创建

### P0-3 定义规则状态流转

目标：

- 不再靠口头描述“这条规则现在算什么状态”

建议状态：

- `draft`
- `in_progress`
- `review`
- `active`
- `rejected`
- `deprecated`

产出物：

- 状态约定文档
- 模板字段对齐

验收线：

- 规则文件中已有 `status`
- 台账口径与规则状态能映射

### P0-4 最终输出单源化整改

目标：

- `review.md`
- Web 页面
- 历史页

全部只消费归一后的最终结果。

核心要求：

- 不再直接拼 topic 原始自由文本
- `excluded_risks` 不得回流到最终报告
- `pending_review_items` 只能以复核项方式输出

产出物：

- 输出装配逻辑调整
- 对应测试

验收线：

- compare 正确时，最终 `review.md` 也正确
- 已剔除误报不再回流到最终成品页

### P0-5 补“最终报告层”测试

目标：

- 测试不仅看 compare
- 还要直接看最终业务成品页

建议补齐：

- 最终 markdown 测试
- Web 卡片标题测试
- 总结区回流阻断测试

验收线：

- 至少有 1 组针对最终报告的专门测试
- 至少有 1 组针对真实文件的成品页测试

### P0-6 建立任务、验收、台账的最小联动

目标：

- 让后续所有规则整改都能追踪

要求：

- 规则任务有编号
- 验收结论回写台账
- 任务状态和台账状态一致

验收线：

- 至少从 `R-001 ~ R-007`、`W-006` 这类已知任务中选一批完成口径统一

## 5. P1 实施清单

P1 重点是把已有资产迁进去。

### P1-1 迁移现有核心规则

建议优先迁移：

- `R-001` 拒绝进口 vs 外标引用
- `R-002` 强制性标准条款未加 ★
- `R-003` 不得将验收方案作为评审因素
- `R-004` 不得将付款方式作为评审因素
- `R-005` 不得提供赠品、回扣或无关商品服务
- `R-006` 不得限定特定专利、商标、品牌、供应商
- `R-007` 不得要求中标人承担验收检测费用

验收线：

- 上述规则均已具备注册表文件
- 规则文件里能查到样本、测试、任务单

### P1-2 建立规则与样本映射

目标：

- 让每条规则都能找到：
  - 正样本
  - 负样本
  - 真实文件回放样本

验收线：

- 每条已激活规则至少有 1 个正样本和 1 个负样本

### P1-3 建立规则与测试映射

目标：

- 每条规则都知道自己对应哪些测试

建议测试分层：

- `unit`
- `replay`
- `report`

验收线：

- 已激活规则都能查到测试文件路径

### P1-4 建立规则与任务单映射

目标：

- 后续任何规则的来源、整改、验收都可追溯

验收线：

- 注册表中能查到对应任务单
- 台账中能反查回规则编号

### P1-5 固化“新规则接入”流程

目标：

- 客户新增规则后，不再临时讨论后直接改代码

应形成标准流程：

1. M 解析规则
2. 建规则草稿
3. 自动生成任务单骨架
4. T 开发
5. M 验收
6. 生效并更新台账

验收线：

- 至少跑通 1 次完整新规则接入演练

## 6. P2 实施清单

P2 再开始做自动化增强。

### P2-1 自动编号

目标：

- 自动生成 `R-xxx`

验收线：

- 新规则创建时不再手工选号

### P2-2 自动生成规则模板

目标：

- 输入规则标题后自动生成规则 YAML 骨架

验收线：

- 新规则创建时不再从空白文件手工写起

### P2-3 自动字段完整性检查

目标：

- 防止缺字段的规则进入 `active`

建议检查：

- 是否有触发条件
- 是否有排除条件
- 是否有输出文案
- 是否有关联样本
- 是否有关联测试

验收线：

- 缺关键字段时，校验失败

### P2-4 自动台账同步

目标：

- 规则状态与台账状态自动对齐

验收线：

- 至少支持提示式同步或脚本式同步

### P2-5 自动生成任务单骨架

目标：

- 新规则分析完成后，自动生成任务单初稿

验收线：

- M 不需要每次从零写任务单

## 7. 推荐优先级排序

按“最值钱、最该先做”的顺序，建议如下：

### 第一优先级

- `P0-4 最终输出单源化整改`
- `P0-5 最终报告层测试`
- `P0-1 建立规则注册表目录`

原因：

- 直接解决当前最痛的“输出错层”和“报告回流”问题

### 第二优先级

- `P0-2 固化规则模板`
- `P0-3 定义状态流转`
- `P1-1 迁移现有核心规则`

原因：

- 让规则治理真正进入仓库资产，而不是继续靠聊天记录

### 第三优先级

- `P1-2 样本映射`
- `P1-3 测试映射`
- `P1-4 任务映射`

原因：

- 形成规则闭环

### 第四优先级

- `P2-*` 自动化增强

原因：

- 这些很有价值，但必须建立在前面骨架已经稳定的基础上

## 8. 第一阶段建议验收线

我建议把“第一阶段完成”的标准定成以下 5 条：

1. 已建立规则注册表目录
2. 已建立规则模板文件
3. `review.md` / Web / history 已统一消费最终分层结果
4. 已有至少 3 条核心规则迁入注册表
5. 已有最终报告层的真实文件自动化测试

只要这 5 条达成，就可以认为：

**V2 规则治理架构已经从“方案阶段”进入“落地阶段”。**

## 9. 推荐执行方式

建议执行时，不要把所有项一次性塞给 T，而是拆成 3 张大任务单：

### Task-G1：治理骨架搭建

内容：

- 规则目录
- 模板
- 状态
- 校验骨架

### Task-G2：输出单源化

内容：

- assembler
- web 输出
- 历史页输出
- 最终报告测试

### Task-G3：规则迁移首批落地

内容：

- 迁移 `R-001 ~ R-003` 或 `R-001 ~ R-007`
- 建样本映射
- 建测试映射

这样比直接下一张超级大单更容易验收。

## 10. 结论

当前项目已经具备进入实施阶段的条件。

后续最重要的不是继续堆零散规则，而是先完成：

1. 规则有地方放
2. 输出有单一来源
3. 误报整改有闭环
4. 验收有统一标准

只要按这份清单推进，V2 就能从“可运行的审查系统”逐步升级成“可持续维护的规则治理系统”。
