# V2 输出治理层架构调整方案

## 1. 背景

近期基于真实文件的连续复核表明，当前系统的主要问题已经不再只是“某条规则没有识别到”，而是出现了更上游、更系统性的输出治理缺口，主要表现为：

1. 同一风险在最终结果中重复输出
2. 已明确应删除的条目未删除
3. 已明确应降级为待补证或排除的条目，仍以正式风险输出
4. 已经做过整改的问题，在真实文件中仍出现回流
5. 规则命中、compare 结果、最终报告、Web 展示之间存在口径漂移

这说明现有 V2 三层能力虽然已经具备：

- 文档解析与结构召回
- 专题识别与候选风险生成
- 规则对比与风险输出

但还缺少一个真正有“最终裁决权”的中间层，去统一处理：

- 风险主键
- 风险合并
- 输出分层
- 弱证据降级
- 误报抑制
- 标题单源化
- 最终输出准入

因此，本方案的目标不是推翻 V2，也不是直接启动 V3，而是在 V2 上补齐一层 `输出治理层`，将当前体系从“能识别风险”提升为“能稳定给出客户可读结果”。

## 2. 结论

从系统架构层面看，`需要调整`。

但调整方式不建议采用“大重构”或“直接起 V3”，而应采用：

`V2 保持主体不变 + 补一层输出治理中枢`

建议对外口径可表述为：

`当前进入 V2.5 阶段，重点补齐输出治理能力，而不是重做识别引擎。`

## 3. 当前根因判断

### 3.1 识别层与输出层耦合过深

当前系统更偏向：

- 专题识别到什么
- compare 命中了什么
- 就直接往最终结果里装配什么

这会导致一个问题：

“识别到了” 不等于 “应该作为正式风险输出”。

### 3.2 缺少统一的风险主键与归并中心

像 `6/7` 这种重复报，本质说明当前系统仍更多依赖：

- 标题相似度
- 来源 topic
- 临时规则判断

而没有统一的风险身份模型。

没有稳定主键，就很难稳定完成：

- 近义项合并
- 历史整改回放
- 同一证据组去重

### 3.3 缺少统一分层裁决器

当前“正式风险 / 待补证 / 已剔除误报”还没有成为真正独立的判定阶段。

于是就会出现：

- 证据弱的项被直接升正式
- 已知模板项没有彻底排除
- 已确认应降级的项又被最终报告拉回

### 3.4 最终输出仍未完全单源化

虽然之前已经推进过输出单源化，但真实文件结果说明还不够硬。

当前仍可能存在：

- topic 原始摘要穿透到最终报告
- 旧标题回流
- compare 已降级，但 report/Web 仍沿用旧风险口径

### 3.5 “整改完成”的门槛定义不够严格

现在容易出现：

- 规则逻辑改了
- 测试加了
- 就判定完成

但真实文件没过、最终页面没对齐，实际上还不能算完成。

## 4. 调整目标

本次架构调整有 5 个核心目标：

1. 把“识别结果”和“最终输出结果”彻底拆层
2. 给每条风险建立稳定身份，支持合并、降级、删除、回放
3. 让正式风险、待补证、已剔除误报三分层成为硬判定，而不是软建议
4. 让 `review.md`、Web、history 页面只消费同一份最终结构化结果
5. 让真实文件回放成为整改完成的必要门禁

## 5. 调整后的目标架构

建议将 V2 调整为以下 6 层：

```text
第1层：文档解析与证据抽取层
第2层：专题识别与候选风险层
第3层：规则命中层
第4层：输出治理层
第5层：最终输出装配层
第6层：监督与治理层
```

其中，真正新增且最关键的是 `第4层：输出治理层`。

## 6. 输出治理层的职责定义

输出治理层位于：

`规则命中层` 之后，`最终输出装配层` 之前。

它不负责重新识别文档，而只负责回答一个问题：

`哪些内容最终可以作为正式风险给客户看，哪些必须降级，哪些必须删除，哪些需要合并。`

### 6.1 风险身份统一

每条候选风险进入输出治理层后，必须先生成稳定的治理主键。

建议至少由以下信息组成：

- `rule_id`
- `risk_family`
- `evidence_anchor`
- `topic_source`
- `document_span`

用途：

- 同题不同写法的归并
- 同一证据组的重复压缩
- 历史整改的精准回放

### 6.2 合并归并

输出治理层必须内建“风险家族”概念。

例如：

- `业绩限定深圳市政府机关`
- `业绩限定特定行政区域`

在业务上应归入同一个风险家族，而不是并列两条正式风险。

输出治理层需要支持：

1. 完全重复合并
2. 近义风险归并
3. 主风险 + 补充说明合并

### 6.3 分层裁决

输出治理层必须是三分层唯一裁决器：

1. `formal_risks`
2. `pending_review_items`
3. `excluded_risks`

不得由：

- topic 层
- 渲染层
- Web 前端

再次自行决定分层。

### 6.4 降级与删除规则

输出治理层必须明确支持：

- 证据不足降级
- 模板区剔除
- 合同范本占位剔除
- 政策条款不完整但证据不足时转待补证
- 已知误报模式直接抑制

也就是说，系统要明确支持：

`命中规则 != 必然正式输出`

### 6.5 标题单源化

当前一个常见问题是：

- 规则命中对了
- 但标题写漂了
- 或同一类风险生成了多个相近标题

因此输出治理层必须统一生成：

- 标准标题
- 风险定性
- 标准整改建议

渲染层不能自己二次改写核心结论。

### 6.6 最终准入控制

只有通过输出治理层准入的对象，才能进入：

- `final_output.json`
- `final_review.md`
- Web 卡片
- 历史页

未通过准入的对象不得以任何形式“夹带回流”。

## 7. 建议新增的内部模块

建议在现有 V2 代码结构中补齐以下模块边界。

### 7.1 Risk Identity Resolver

职责：

- 为候选风险生成稳定主键
- 建立风险家族
- 支持跨 topic 对齐

### 7.2 Risk Merge Resolver

职责：

- 检测完全重复
- 识别近义重复
- 选择主标题
- 聚合证据与补充说明

### 7.3 Risk Layer Decider

职责：

- 判定正式风险 / 待补证 / 已剔除误报
- 处理降级和删除逻辑
- 承接模板、证据不足、例外条款等规则

### 7.4 Risk Canonicalizer

职责：

- 标题单源化
- 风险等级单源化
- 整改建议单源化

### 7.5 Output Admission Gate

职责：

- 统一控制哪些内容能进入最终业务输出
- 阻止 topic 原始自由文本直接穿透到最终报告

## 8. 与现有 V2 的衔接方式

本方案不要求推翻现有三层，而是按“外挂治理层”的方式接入。

衔接方式建议如下：

### 8.1 保留现有解析与专题能力

以下能力原则上保留：

- 文档解析
- 结构召回
- 专题识别
- compare / rule 命中

### 8.2 在 compare 之后新增治理收口

当前更合理的链路应变为：

```text
document parse
-> topic reviews
-> compare / rule hits
-> output governance
-> final output
```

### 8.3 输出层只读治理结果

`review.md`、Web、history 页面应只读：

- 输出治理层产出的最终结构化对象

而不是再读：

- topic summary
- baseline 原始摘要
- 旧式 report 拼接文本

## 9. 必须建立的硬性门禁

这次架构调整如果只是“多一个模块”还不够，必须同步增加硬门禁。

### 9.1 真实文件回放门禁

以后任务通过，至少要满足：

1. 单元测试通过
2. 回归样本通过
3. 指定真实文件回放通过
4. 最终报告和 Web 展示一致

缺一不可。

### 9.2 去重门禁

对于已知重复家族，必须增加自动断言：

- 同一真实文件结果中，不允许并列输出两条近义正式风险

### 9.3 降级门禁

对于已知需降级或排除的场景，必须增加自动断言：

- 不能继续进入 `formal_risks`

### 9.4 标题漂移门禁

同一规则家族输出标题应固定，不得每次复跑生成不同标题。

## 10. 管理流程上的同步调整

除了代码架构，还需要同步调整管理口径。

### 10.1 一条任务不再默认“改完规则就算完成”

以后必须区分：

- `R-xxx`
  - 规则识别能力补齐
- `W-xxx`
  - 输出治理、真实文件闭环、Web 对齐

规则补齐不等于客户结果可用。

### 10.2 关单条件升级

任务要正式关单，建议至少满足：

1. 规则已注册
2. 样本已补
3. 测试已补
4. 真实文件已复跑
5. 输出治理结果正确
6. Web 展示正确

### 10.3 误报和漏报都进入治理资产

以后误报、漏报都应沉淀为：

- 任务单
- 风险家族
- 正负样本
- 回归用例
- 真实文件基线

## 11. 近期优先级建议

如果按价值排序，我建议优先做 3 件事：

### P0：输出治理层骨架

先把：

- 风险身份
- 合并
- 分层
- 准入

四个骨架补起来。

### P1：真实文件门禁化

将当前柴油发电机组文件这类真实案例纳入固定回放门禁。

### P2：风险家族与标题单源化

把高频问题先纳入固定家族：

- 重复业绩限定
- 检测费用打包
- 履约保证金超标
- 项目负责人评分项
- 外标引用与非进口矛盾

## 12. 不建议的路线

当前不建议：

1. 直接起全新 V3
2. 继续只做点状补规则
3. 让渲染层继续二次拼装自由文本
4. 把“人工验收结论”停留在聊天记录，不沉淀为治理资产

## 13. 管理层汇报口径

可以对领导或技术负责人这样汇报：

当前系统识别能力已经具备基础，但真实文件连续复核表明，主要短板已从“能不能识别到”转为“能不能稳定、唯一、准确地输出给客户看”。因此项目下一阶段不建议推翻重做，而是在现有 V2 上补齐输出治理层，重点解决重复报、误报降级失败、标题漂移、旧口径回流和真实文件闭环不稳定的问题。该调整属于架构收口和治理增强，目标是把系统从“可用”推进到“可控、可验收、可持续演进”。
