# 任务单 / 验收单模板规范

## 1. 目标

本规范用于统一：

- 任务单格式
- T 回传格式
- M 验收格式
- 状态更新口径

适用范围：

- `R-xxx`
- `E-xxx`
- `W-xxx`
- `P-xxx`
- `Q-xxx`
- `O-xxx`

## 2. 任务单模板

以下为标准任务单模板。

```md
# <任务编号> 任务单

## 基本信息

- 任务编号：`<任务编号>`
- 任务名称：<任务名称>
- 任务状态：`待下发`
- 下发对象：`T`
- 监督角色：`M`
- 所属域：`<product/... 或 governance/... 或 quality/...>`
- 任务类型：`<规则整改 / 样本整改 / Web整改 / 产品功能 / 质量治理 / 运营支持>`

## 任务背景

<描述客户提出的规则、真实文件场景、当前问题现状>

## 整改目标

<说明本次要达到什么结果>

## 标准口径

### 风险标题

`<标准风险标题>`

### 问题定性

`<高风险 / 中风险 / 低风险 / 需人工复核>`

### 审查类型

`<标准审查类型>`

### 风险判断要求

- <判断条件1>
- <判断条件2>
- <判断条件3>

### 整改建议要求

- <建议1>
- <建议2>

## 具体整改要求

### 1. 规则 / 功能整改

- <要求1>
- <要求2>

### 2. 样本整改

- 正样本：<...>
- 负样本：<...>
- 真实文件样本：<...>

### 3. 测试整改

- <规则抽取测试>
- <compare 测试>
- <真实文件回放测试>
- <Web 展示测试>

## 交付物要求

T 提交时必须附：

1. 改了哪些文件
2. 新增了哪些规则 / 样本 / 测试
3. 跑了哪些命令，结果如何
4. 新结果目录
5. Web 链接
6. 自评是否过线

## M 验收标准

### 通过线

- <通过条件1>
- <通过条件2>

### 不过线情形

- <不过线情形1>
- <不过线情形2>

## T 回复格式

1. 本次补的是哪类问题
2. 改了哪些文件
3. 新增了哪些规则 / 样本 / 测试
4. 跑了哪些命令，结果如何
5. 新结果目录是什么
6. Web 链接是什么
7. 自评是否已过线
```

## 3. T 回传模板

T 回传时统一使用以下结构：

```md
## 回传信息

1. 本次补的是哪类问题
2. 改了哪些文件
3. 新增了哪些规则与结构化字段
4. 新增了哪些样本与测试
5. 跑了哪些命令，结果如何
6. 新结果目录是什么
7. Web 链接是什么
8. 自评是否已过线
```

## 4. M 验收单模板

以下为标准验收单模板。

```md
## <任务编号> 验收结论

- 验收结果：`通过 / 未通过`
- 验收评分：`<分数>/100`

### 主要结论

- <通过项或问题项1>
- <通过项或问题项2>
- <通过项或问题项3>

### 证据

- 结果目录：<路径>
- comparison.json：<路径>
- review.md：<路径>
- Web 链接：<链接>
- 测试命令：<命令>
- 测试结果：<结果>

### 扣分项

- <扣分项1>
- <扣分项2>

### 最终判断

- <是否达到通过线>
- <是否需要补强>
- <是否可以关单>
```

## 5. 台账状态更新规范

M 在不同阶段需同步更新台账状态：

| 阶段 | 台账状态 |
| --- | --- |
| 规则已明确，尚未下发 | `待下发` |
| T 已开始处理 | `整改中` |
| T 已回传等待验证 | `待验收` |
| M 验收通过 | `已通过` |
| M 验收未通过 | `未通过` |
| M 明确结束跟踪 | `已关闭` |

## 6. 关单规范

当用户明确说：

- `关单`
- `执行关单`
- `正式关单`

M 必须同步完成：

1. 更新总台账状态为 `已关闭`
2. 更新任务详情状态为 `已关闭`
3. 更新当前结论为 `已完成验收并正式关单`

## 7. 评分建议规范

建议统一使用 `100` 分制。

推荐口径：

- `95~100`：已过线，可直接通过
- `85~94`：基本完成，但有明显管理项或展示层尾差
- `70~84`：核心功能已部分完成，但仍有实质缺口
- `<70`：未达可用标准

## 8. 使用建议

建议后续所有任务都按以下顺序执行：

1. M 生成任务单
2. 台账挂项
3. T 开发并回传
4. M 验收评分
5. 台账更新状态
6. 用户确认后关单
