# 候选规则接入与纳管迁移治理方案

## 1. 文档目标

本文档用于定义“外部候选规则接入本项目”的标准治理方案。

本方案聚焦解决以下问题：

1. 外部系统提供了大量审查点，但不能直接导入当前正式规则库。
2. 当前项目的正式规则数量较少，确实需要持续补充，但补充过程必须可控。
3. 新规则接入过程中，必须避免污染现有运行链路，避免一次性批量引入误报。
4. 规则新增、误报整改、漏报补强、任务下发、验收关单，需要进入统一治理闭环。

本方案的目标不是“快速堆规则数量”，而是建立一套稳定的候选规则纳管机制，持续提升真实文件风险点识别的准确率。

## 2. 结论摘要

针对“外部 150 条通用审查点是否应补入本项目规则库”的判断如下：

1. 这批审查点有价值，适合作为候选规则池。
2. 这批审查点不适合直接批量导入 `rules/registry/`。
3. 当前最优路径不是“全量规则化”，而是先建立候选池、分流机制、纳管门禁与回滚机制。
4. 本次工作应作为 `V2` 的治理增强实施，不建议新开 `V3`。

原因在于，这批审查点本质上混合了多种类型：

1. 可直接落地为正式规则的稳定禁限项。
2. 依赖具体品类、区域或采购画像的条件型规则。
3. 需要上游抽取、分类、场景理解支撑的能力型需求。
4. 不适合纳入当前项目、需要删除或暂缓的条目。

因此，项目需要先建立“候选层 -> 注册层 -> 运行层”的分层治理结构。

## 3. 治理目标与边界

### 3.1 核心治理目标

本方案需同时满足以下四个目标：

1. 将外部审查点统一接入项目内部的候选规则池。
2. 保证任何外部条目不能绕过审核直接污染正式规则注册表。
3. 让每一条规则的接入、实现、验收、回写都可追踪。
4. 让失败批次可快速回滚，避免影响已有稳定识别能力。

### 3.2 本次不做的事情

本方案明确不包括以下动作：

1. 不把 150 条候选点一次性全部转成 `R-xxx`。
2. 不绕过任务单、验收单直接修改正式规则运行逻辑。
3. 不把能力型诉求强行包装成正式规则。
4. 不把“补规则”升级成对当前 V2 审查架构的整体重构。

### 3.3 成功标准

本方案通过的标准是：

1. 候选规则能够被统一收纳、编号、追踪。
2. 正式规则库仍保持干净，只包含经过验收的规则。
3. 任意一批规则迁移失败后，可以定位到批次并执行回滚。
4. 后续新增客户规则时，可以复用同一套治理入口，而不是重新临时组织。

## 4. 三层规则资产结构

### 4.1 候选层

候选层用于承接外部导入的原始审查点。

其特点如下：

1. 允许条目尚未完全规范。
2. 允许混杂正式规则候选、条件型条目、能力型条目和待删除项。
3. 不直接参与线上正式判定。
4. 主要承担留档、分流、追踪、映射职责。

### 4.2 注册层

注册层是项目的正式规则注册表。

只有在以下条件全部满足后，候选条目才可进入注册层：

1. 已有明确的规则编号。
2. 已补齐触发条件。
3. 已补齐排除条件或负样本条件。
4. 已确定标准风险文案。
5. 已确定建议整改文案。
6. 已明确应落层级。
7. 已补齐样本、测试和任务单映射。
8. 已通过 M 验收。

注册层是当前项目唯一正式规则源。

### 4.3 运行层

运行层是代码和产品实际消费的规则输出层。

运行层只允许消费以下资产：

1. 已通过验收的正式规则。
2. 已验证可用的正式样本和测试。
3. 已完成输出层联动校验的标准文案。

运行层不得直接消费候选池原始内容。

## 5. 目录结构建议

建议在 `rules/` 下增加候选规则治理区域，与正式注册表物理隔离：

```text
rules/
├── registry/                  # 正式规则注册表，只放已纳管规则
├── candidates/                # 候选规则池
│   ├── sources/               # 外部来源原件或摘录副本
│   ├── imports/               # 标准化导入结果
│   ├── mappings/              # 候选条目分流台账
│   └── snapshots/             # 导入与分流快照
├── templates/
└── schemas/
```

各目录职责如下：

### 5.1 `rules/candidates/sources/`

用于保存外部规则来源的副本或标准化摘录。

特点：

1. 只做留档和追溯。
2. 不直接参与运行。
3. 用于后续核对原始语义和来源可信度。

### 5.2 `rules/candidates/imports/`

用于保存某次导入后的结构化候选清单。

特点：

1. 可记录某个来源在某个时间点的标准化版本。
2. 便于后续比较不同导入批次的变化。
3. 是候选规则台账的上游输入。

### 5.3 `rules/candidates/mappings/`

用于保存候选规则分流台账。

这是本机制的核心数据层，承担以下职责：

1. 记录候选条目的治理判断。
2. 记录是否建议纳入当前项目。
3. 记录是否已生成任务单。
4. 记录是否已完成迁移和验收。

### 5.4 `rules/candidates/snapshots/`

用于保存每次导入、分流或迁移前后的快照。

这是回滚和问题定位的关键资产。

## 6. 候选规则分流台账设计

建议建立一份标准化候选规则分流台账，至少包含以下字段：

| 字段 | 含义 |
| --- | --- |
| `candidate_id` | 候选编号，例如 `CR-001` |
| `source_name` | 来源文档名称 |
| `source_rule_text` | 原始审查点文本 |
| `source_category` | 来源专题或家族 |
| `decision` | 分流结论：`formal_rule / conditional_rule / capability_item / drop` |
| `decision_reason` | 分流原因说明 |
| `target_rule_id` | 若纳管，映射到的正式规则编号 |
| `target_layer` | 建议落层：正式风险 / 待补证 / 排除 |
| `profile_dependency` | 是否依赖品类、地区、采购画像等前置条件 |
| `negative_conditions` | 是否已定义负样本或排除条件 |
| `samples_status` | 样本状态：无 / 已补 / 已验证 |
| `tests_status` | 测试状态：无 / 已补 / 已通过 |
| `task_id` | 关联任务单编号 |
| `status` | 当前状态：`new / triaged / in_progress / accepted / rejected / migrated` |
| `snapshot_id` | 属于哪一批导入或分流快照 |

### 6.1 候选条目分流结论定义

分流结论统一使用以下四类：

1. `formal_rule`
   - 可直接进入正式规则纳管流程
   - 但仍需补任务、样本、测试与验收
2. `conditional_rule`
   - 具备规则价值，但依赖特定场景、品类、地区或画像
   - 不宜直接作为通用规则启用
3. `capability_item`
   - 不能靠单条规则稳定解决
   - 需要抽取、结构化理解、主题识别或专题能力建设承接
4. `drop`
   - 当前不纳入项目
   - 包括待删除项、模板残留、价值低或边界不清的条目

### 6.2 分流判定标准

候选条目要进入 `formal_rule`，至少应满足：

1. 能脱离单一文件个案独立成立。
2. 能写出稳定的触发条件。
3. 能写出明确的排除条件或负样本边界。
4. 能形成统一标准风险文案。
5. 能形成统一建议整改文案。
6. 能设计回归样本和测试闭环。

只要以上任一条件明显不满足，就不应直接进入正式规则。

## 7. 回滚与备份机制

### 7.1 回滚是硬约束

候选规则接入必须以“失败可恢复”为前提设计。

不能接受以下风险：

1. 一批候选规则导入后无法区分是哪批造成误报。
2. 一次错误迁移污染了正式规则库，但没有恢复点。
3. 候选池、任务单、注册表、测试之间无法对齐，导致无法追责。

### 7.2 快照批次机制

建议每次关键动作都生成 `snapshot_id`，例如：

1. `SNAP-2026-04-07-import-001`
2. `SNAP-2026-04-08-triage-001`
3. `SNAP-2026-04-10-migrate-001`

每个快照至少记录：

1. 当次候选池输入文件。
2. 当次分流台账版本。
3. 当次计划迁移的规则清单。
4. 对应任务单列表。
5. 对应样本与测试状态摘要。
6. 对应验收结论摘要。

### 7.3 回滚策略

回滚按“批次”执行，而不是按零散文件临时处理。

建议回滚规则如下：

1. 若某批规则迁移引入明显误报，优先回退该迁移批次。
2. 若只是候选池分流判断错误，回退候选映射快照，不影响正式规则运行。
3. 若正式规则已入库但验收后发现问题，转补强任务单或按批次回退该规则启用状态。

### 7.4 安全原则

为保证迁移安全，必须遵守以下原则：

1. 候选池与正式规则注册表物理隔离。
2. 正式纳管采用小批量增量迁移。
3. 每批迁移均保留前后状态记录。
4. 先验证再扩大，不做一次性全量切换。

## 8. 标准流转流程

建议将候选规则治理流程固定为以下七步：

### 8.1 候选接入

外部规则先进入 `rules/candidates/sources/` 与 `rules/candidates/imports/`。

此阶段只做接入和结构化，不进入正式规则库。

### 8.2 M 分流判定

由 M 对候选条目逐条判定：

1. 是否可成为正式规则。
2. 是否只是条件型规则。
3. 是否应进入能力建设 backlog。
4. 是否应删除或暂不纳管。

产出物：

1. 候选规则分流台账更新。
2. 分流批次快照。

### 8.3 首批纳管立项

从 `formal_rule` 候选中挑出高优先级条目，生成正式任务单，例如：

1. `R-009`
2. `R-010`

并同步回写总台账。

### 8.4 T 实施落地

T 只处理已经立项的候选条目，不处理整个候选池。

T 的实施内容包括：

1. 补正式规则注册表。
2. 补代码逻辑。
3. 补样本。
4. 补测试。
5. 补标准文案。
6. 补输出层联动。

### 8.5 M 验收

M 按以下维度验收：

1. 规则字段完整性。
2. 标准文案完整性。
3. 样本是否覆盖正负例。
4. 测试是否通过。
5. 真实文件回放是否符合预期。
6. 输出层是否与规则结论一致。

通过后，候选条目状态从 `in_progress` 更新为 `migrated`。

### 8.6 运行观察

规则正式启用后，继续使用真实文件与回归集观察：

1. 是否出现误报。
2. 是否出现漏报。
3. 是否出现错层。
4. 是否出现文案错误。
5. 是否出现 Web 展示偏差。

若出现问题，转入 `W-xxx` 或 `R-xxx-F` 补强闭环。

### 8.7 关单与回写

每次纳管完成后，应同步更新：

1. 候选规则分流台账。
2. 正式规则注册表。
3. 总整改台账。
4. 对应任务单状态。
5. 验收记录或回放记录。

## 9. M / T / 自动化职责边界

### 9.1 M 的职责

M 负责：

1. 判断候选条目是否纳管。
2. 判断落正式规则还是能力项。
3. 生成任务单。
4. 执行验收。
5. 执行关单与状态回写。
6. 判断误报、漏报是否成立。

### 9.2 T 的职责

T 负责：

1. 根据已批准任务单实现规则。
2. 补规则文件。
3. 补代码逻辑。
4. 补样本和测试。
5. 补输出层联动。
6. 提交验收材料和回放证据。

T 不负责直接决定候选条目是否纳入正式规则库。

### 9.3 自动化职责

自动化适合承担以下辅助工作：

1. 校验候选台账字段是否完整。
2. 校验规则编号、任务单、台账状态是否一致。
3. 生成快照摘要。
4. 生成初始规则骨架或任务草稿。
5. 提醒缺样本、缺测试、缺排除条件的条目。

自动化不替代 M 的治理判断。

## 10. 版本策略

本方案定位为 `V2` 的治理增强，不建议另起 `V3`。

理由如下：

1. 当前项目已有可运行的 V2 三层审查链路。
2. 当前缺口在于规则治理入口、候选池管理和误报回滚机制，而不是底层审查能力整体重做。
3. 若此时新开 `V3`，会把治理问题扩大成系统重构问题，增加失控风险。

因此，建议命名为：

1. `V2 治理增强`
2. 或 `V2 Rule Governance Extension`

## 11. 实施顺序建议

建议按以下四步推进：

### 11.1 第一步：搭建候选池与分流台账

目标：

1. 先把外部 150 条审查点接入项目内部治理体系。
2. 建立统一候选编号、来源记录和分流状态。
3. 暂不改动正式规则运行链路。

这是最安全、最必要的一步。

### 11.2 第二步：首批高价值规则纳管

目标：

1. 从分流结果中挑选高频、高价值、确定性强的条目。
2. 首批建议控制在 `10 ~ 20` 条。
3. 重点追求“准”，而不是“多”。

### 11.3 第三步：补纳管校验与回滚机制

目标：

1. 形成快照管理。
2. 形成编号追踪。
3. 形成规则、任务单、样本、测试的一致性检查。

### 11.4 第四步：再做半自动化提效

目标：

1. 自动生成候选骨架。
2. 自动生成任务草稿。
3. 自动检查缺样本、缺测试、缺负样本边界。

这一步属于提效，不应阻塞前面三步。

目标：

1. 接住全部外部候选条目。
2. 建立分流台账。

1. 建立快照批次机制。
2. 建立迁移前后状态留档机制。

### 12.3 `Task-G8：首批高价值规则纳管`

## 14. 最终结论

外部 150 条审查点不应直接批量转成正式规则。

当前正确路径应为：

1. 先接入候选池。
2. 再做分流。
3. 再择优纳管。
4. 再用真实文件和回归样本验证。
5. 出现误报、漏报后继续走 M/T 闭环整改。

本方案的核心价值不在于“规则数量增加”，而在于把外部规则扩充、客户反馈、误报整改、任务下发、验收关单全部纳入统一的治理结构中，让项目后续持续演进时不再失控。
