# 项目测试角色职责与协作机制 V1

## 1. 文档目的

本文件用于在 `test_getst` 现有 `M-管理`、`O-运营`、`T-开发` 之外，补充 `Q-测试` 角色，明确：

- 测试在本项目中的职责边界
- 测试如何与 `M / O / T` 协作
- 什么问题由测试发现、拦截、回归、放行
- 当前项目的测试基线与优先事项

本文件不替代已有角色文档，而是作为测试补充规范，与 [project-roles-and-issue-closure-v1.md](https://github.com/zeranlin/agent_bid_check/blob/main/docs/governance/project-roles-and-issue-closure-v1.md) 并行使用。

## 2. 项目理解

`test_getst` 是一套招标文件智能审查系统，当前核心目标是围绕 `V2` 链路持续提升：

- 合规风险识别能力
- 证据可追踪能力
- 结果可解释能力
- 真实文件场景下的稳定性

当前主链路为：

1. 文本抽取
2. 全文基线审查
3. 结构增强
4. 证据召回
5. 专题深审
6. 汇总对比
7. 最终报告装配
8. Web / CLI 展示与结果落盘

测试工作的核心，不只是验证“程序能跑通”，还要验证：

- 规则识别是否准确
- 负样本是否被误伤
- 真实文件回放是否稳定
- Web 展示与落盘产物是否完整
- 新规则、新专题、新 compare 规则是否引入回归

## 3. Q-测试角色定位

### 3.1 角色定义

`Q-测试` 负责项目质量基线的建立、执行、回归、阻断与结果回传，是连接业务风险、规则治理和技术实现的质量守门角色。

### 3.2 角色目标

`Q-测试` 的目标不是替 `T-开发` 写功能，也不是替 `M-管理` 定优先级，而是确保：

- 每次改动可验证
- 每类问题可复现
- 每条规则有样本支撑
- 每个高风险问题有回归覆盖
- 每次上线前有清晰放行依据

## 4. Q-测试职责清单

### 4.1 测试基线维护

- 维护 `pytest` 自动化测试基线。
- 维护 `tests/` 下的单测、集成测试、真实文件回放测试、Web 兼容测试。
- 维护 `data/examples/`、`data/eval/` 下与回归相关的样本质量。

### 4.2 回归验证

- 对 `V2` 主链路执行回归验证，覆盖：
  - `extract_document_text`
  - `build_structure_map`
  - `build_evidence_map`
  - `run_topic_reviews`
  - `compare_review_artifacts`
  - `assemble_v2_report`
  - `save_review_artifacts_v2`
- 对 Web 入口执行兼容验证，重点关注：
  - 任务创建
  - 状态流转
  - 历史结果加载
  - 视图兼容

### 4.3 缺陷发现与复现

- 将线上反馈、评测失败、真实文件异常转成可复现问题。
- 为每类问题补齐最小复现样本。
- 明确问题属于：
  - 误报
  - 漏报
  - 规则边界错误
  - 结构召回错误
  - compare 联动错误
  - 展示兼容错误
  - 配置或运行错误

### 4.4 质量门执行

- 对 `P0 / P1` 问题建立阻断机制。
- 对新增规则或整改任务要求同时补：
  - 正样本
  - 负样本
  - 必要时真实文件样本
  - 对应自动化测试
- 对“仅改逻辑、不补测试”的交付保持否决权建议。

### 4.5 结果验收支持

- 为 `M-管理` 提供测试结论、风险说明、未过线项。
- 为 `O-运营` 提供复测口径和对外可说明的验证结论。
- 为 `T-开发` 提供失败样本、失败断点、回归范围。

### 4.6 测试资产沉淀

- 建立问题到样本、样本到测试、测试到规则的映射。
- 推动高频问题沉淀为长期回归集。
- 推动真实文件问题沉淀为 `W-xxx / Q-xxx / R-xxx` 级别测试资产。

## 5. Q 与 M / O / T 的协作关系

### 5.1 与 M-管理协作

`M` 负责定优先级、定排期、定验收标准，`Q` 负责把“是否过线”转成可执行测试口径。

`Q` 需要向 `M` 输出：

- 当前测试基线是否通过
- 哪些问题阻断上线
- 哪些问题属于可带病运行
- 哪些任务缺少测试交付物

### 5.2 与 O-运营协作

`O` 负责收集客户问题，`Q` 负责把问题转成可复测案例。

`Q` 需要向 `O` 说明：

- 需要补哪些上下文
- 是否已经复现
- 属于哪一层问题
- 修复后应该如何复测

### 5.3 与 T-开发协作

`T` 负责实现和修复，`Q` 负责验证修复是否真正生效且没有引入新回归。

`Q` 对 `T` 的配合要求：

- 修复必须附带最小复现样本或指向已有样本
- 修复必须说明影响范围
- 修复必须附带对应测试或明确为何暂不适合自动化

## 6. Q-测试边界

- 不替 `M` 决定需求优先级和上线策略。
- 不替 `O` 对外承诺交付时间。
- 不替 `T` 完成功能实现。
- 发现风险后必须给出明确测试结论，但不直接拍板业务范围。

## 7. 测试工作流

### 7.1 问题进入

问题来源包括：

- `O` 提交的客户反馈
- `M` 下发的整改任务
- `T` 提交后的自测回归
- `pytest` 自动化失败
- 真实文件回放失败
- Web 展示异常

### 7.2 Q 处理动作

`Q` 收到问题后，至少完成以下动作：

1. 判断是否可复现
2. 判断影响层级和风险等级
3. 补充或绑定样本
4. 标记应进入的测试层
5. 回传结论给 `M / O / T`

### 7.3 Q 输出格式

测试回传至少包含：

1. 问题编号或样本编号
2. 复现方式
3. 实际结果
4. 期望结果
5. 影响范围
6. 是否阻断上线
7. 是否已补测试
8. 回归结果

## 9. 当前质量现状

本次执行 `pytest -q` 的结果为：

- `119` 通过
- `2` 失败

当前失败点如下：

1. `tests/test_v2_topics_eval.py`
   - `topic_technical_standard_background_negative_004`
   - 现象：项目背景中的外标介绍被误识别为技术参数外标约束
   - 风险：会把背景描述误判成技术限制，产生误报

2. `tests/test_v2_topics_eval.py`
   - `topic_scoring_acceptance_plan_negative_contract_only_006`
   - 现象：纯评分描述中没有“验收方案禁入评分”规则句，却被打上规则命中
   - 风险：会把普通评分文本误判为规则冲突，扩大误报面

这两类问题都属于当前阶段必须优先盯防的“负样本边界漂移”问题。

## 10. Q-测试当前优先职责

现阶段 `Q` 的首要职责按优先级排序如下：

1. 盯住 `V2` 评测红线，先把自动化回归恢复到全绿。
2. 重点补强“背景介绍 / 规则说明 / 评分正文”之间的边界负样本。
3. 对真实文件高频问题建立回放样本，避免只在样例中通过。
4. 对 `T` 每次整改要求补对应自动化测试。
5. 对 `M` 提供明确放行结论，不接受“感觉差不多可以上线”。

## 11. 与各角色的任务接口

### 给 M-管理

`Q` 每轮应提交：

- 当前测试通过率
- 阻断问题清单
- 可延期问题清单
- 建议验收结论

### 给 O-运营

`Q` 每轮应提交：

- 可复测的问题列表
- 复测步骤
- 修复前后差异说明
- 仍需客户确认的边界项

### 给 T-开发

`Q` 每轮应提交：

- 失败测试名
- 对应样本编号
- 失败现象
- 建议补测点
- 回归范围

## 12. 放行建议

当前项目在进入新一轮版本验收前，建议执行以下放行条件：

1. `pytest` 全量通过。
2. `V2` 主题评测、compare 评测、真实文件回放无新增回退。
3. 新增规则必须同时具备正负样本。
4. 影响 Web 展示的改动必须经过 `test_v2_web_compat.py` 验证。
5. `P0 / P1` 问题未关闭前，不建议放行。

## 13. 结论

从当前项目状态看，测试角色不应只是“最后跑一下命令的人”，而应承担以下三项核心责任：

- 把问题变成可复现样本
- 把修复变成可回归测试
- 把上线变成有证据支撑的放行决定

后续建议正式将 `Q-测试` 纳入项目协作体系，与 `M / O / T` 形成：

`O 收集问题 -> M 定级排期 -> T 实现修复 -> Q 验证放行 -> O 回传客户`
