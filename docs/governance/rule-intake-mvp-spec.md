# 规则接入 MVP Spec

## 1. 文档目标

本 spec 用于定义“规则接入 MVP”的最小可用版本，目标是在不做大规模项目重构的前提下，让“客户提出新规则”这件事能够进入标准治理闭环。

本 spec 关注：

- 新规则如何登记
- M 如何分析归类
- 如何生成任务编号与任务单
- T 如何按固定格式回传
- M 如何验收并入库
- 如何把新规则纳入台账、规则总目录和真实文件回归

本 spec 不追求一次性做成完整平台，而是先做出可执行、可复用、可扩展的最小治理闭环。

## 2. MVP 目标

规则接入 MVP 必须解决以下 6 个问题：

1. 客户提出新规则时，有统一登记入口
2. M 能将规则转成标准治理对象
3. 每条规则都有唯一编号和任务单
4. T 的交付格式固定，减少反复沟通成本
5. M 的验收依据固定，状态更新一致
6. 通过的规则能进入规则总目录，并纳入回归基线

一句话目标：

`把“客户提新规则 -> M分析 -> T开发 -> M验收 -> 规则入库”做成项目内的标准流程。`

## 3. 非目标

MVP 阶段不做以下事项：

1. 不做新的 Web 页面或管理后台
2. 不做自动编号服务
3. 不做数据库化规则平台
4. 不做复杂权限管理
5. 不做覆盖率仪表盘
6. 不做项目目录的全量迁移重构

说明：

MVP 阶段优先采用“文档 + 台账 + 模板 + 现有仓库流程”落地。

## 4. 用户与角色

### 4.1 客户 / 业务

职责：

- 提供新规则
- 提供真实文件或原文场景
- 说明业务为什么认为存在风险

### 4.2 M

职责：

- 规则登记
- 分析归类
- 生成编号与任务单
- 更新台账
- 验收评分
- 规则入库

### 4.3 T

职责：

- 实现规则
- 补样本
- 补测试
- 跑回归
- 回传结果

## 5. MVP 范围

MVP 阶段只覆盖以下治理对象：

1. `R-xxx`
   - 新规则识别整改
2. `E-xxx`
   - 样本 / 评测补强
3. `W-xxx`
   - Web / 展示 / 链路补强

MVP 阶段不强制覆盖：

- `P-xxx`
- `Q-xxx`
- `O-xxx`

这些可在后续扩展。

## 6. 输入与输出

## 6.1 输入

一条新规则进入系统时，最少要有以下输入：

1. 规则原文
2. 风险场景原文
3. 来源文件或来源链接
4. 业务为什么认为存在风险

推荐补充信息：

5. 优先级
6. 是否已有历史结果链接
7. 是否已有相近规则

## 6.2 输出

MVP 阶段要求产出以下对象：

1. 规则登记记录
2. 任务编号
3. 任务单
4. 台账记录
5. 验收结论
6. 规则总目录条目

## 7. 规则对象模型

MVP 阶段每条规则最少维护以下字段：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 规则名称 | 是 | 标准化规则名称 |
| 规则原文 | 是 | 客户提出的新规则原文 |
| 风险场景原文 | 是 | 对应的真实文件命中原文 |
| 所属专题 | 是 | 如 `policy / scoring / acceptance / technical_standard` |
| 问题类型 | 是 | 规则整改 / 样本整改 / 展示整改 |
| 标准风险标题 | 是 | 对外统一风险标题 |
| 审查类型 | 是 | 标准审查类型 |
| 风险等级 | 是 | 高 / 中 / 低 / 需人工复核 |
| 规则码 | 否 | compare 规则码，新增规则时允许为空，验收通过后补齐 |
| 对应任务编号 | 是 | 如 `R-008` |
| 当前状态 | 是 | 待下发 / 整改中 / 待验收 / 已通过 / 未通过 / 已关闭 |
| 样本映射 | 否 | 对应样本文件 |
| 测试映射 | 否 | 对应测试文件 |
| 真实文件基线 | 否 | 对应真实文件回放记录 |

## 8. 状态机

规则接入 MVP 使用以下状态流转：

```text
待登记 -> 待下发 -> 整改中 -> 待验收 -> 已通过 -> 已关闭
                              \-> 未通过 -> 整改中
```

状态定义：

- `待登记`：规则已被口头提出，但未形成治理对象
- `待下发`：已完成分析和编号，尚未正式给 T
- `整改中`：T 已开始实现
- `待验收`：T 已回传，等待 M 复核
- `已通过`：M 判定已过线
- `未通过`：M 判定未过线，需要继续整改
- `已关闭`：任务通过验收并确认停止跟踪

## 9. MVP 流程

### 步骤 1：客户提出新规则

客户以自然语言提出：

- 新规则原文
- 真实文件原文
- 风险判断

### 步骤 2：M 规则登记

M 先判断：

1. 是否已有规则覆盖
2. 是新规则，还是已有规则补强
3. 属于 `R / E / W` 哪一类

### 步骤 3：M 生成标准口径

M 需要产出：

1. 标准风险标题
2. 审查类型
3. 风险判断要求
4. 整改建议要求
5. 样本要求
6. 测试要求

### 步骤 4：M 生成任务单与台账

M 需要同步完成：

1. 分配任务编号
2. 生成任务单
3. 更新台账状态

### 步骤 5：T 开发与回传

T 按固定模板回传：

1. 改了哪些文件
2. 新增了哪些规则与结构化字段
3. 新增了哪些样本与测试
4. 跑了哪些命令
5. 新结果目录
6. Web 链接
7. 自评是否过线

### 步骤 6：M 验收

M 按统一模板验收，判断：

1. 是否命中真实场景
2. 是否误报
3. 是否生成标准规则码
4. Web 是否一致
5. 是否纳入回归

### 步骤 7：规则入库

验收通过后：

1. 更新规则总目录
2. 更新任务状态
3. 纳入真实文件基线

## 11. MVP 验收标准

规则接入 MVP 通过的标准是：

1. 新规则提出后，M 可以在一次对话内完成：
   - 规则归类
   - 任务编号
   - 任务单生成
   - 台账挂项
2. T 可以按模板回传，不需要反复确认格式
3. M 可以按模板完成验收、打分、关单
4. 通过后的规则可以进入规则总目录
5. 整个流程不依赖额外系统页面，也能稳定运行

## 12. 风险与约束

MVP 阶段主要风险：

1. 规则编号仍以人工分配为主
2. 台账与规则库更新仍靠人工维护
3. 真实文件基线还未完全平台化
4. 目录仍存在历史文件与新治理文档并存的情况

对应策略：

1. 先保证流程稳定
2. 再逐步产品化

## 13. 后续扩展方向

MVP 稳定后，下一阶段可以扩展：

1. 规则接入表单化
2. 自动生成任务单草稿
3. 规则总目录正式化
4. 真实文件基线库
5. 覆盖率统计与回归门禁

## 14. 建议的下一步

本 spec 通过后，下一步建议直接拆成执行清单，顺序如下：

1. 固化规则登记模板
2. 固化规则总目录正式版
3. 固化台账更新规范
4. 选一条新规则，按 MVP 流程完整跑一遍
5. 输出第一版“规则接入执行清单”
