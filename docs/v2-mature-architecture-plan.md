# 成熟版 V2 架构方案

## 1. 目标定位

当前 V2 已具备三层链路的基本能力：

- 第一层：全文基线审查
- 第二层：结构增强
- 第三层：专题深审

但当前版本仍属于“能用版三层”，还不是“成熟版三层”。成熟版 V2 的建设目标是：

- 提高章节与证据召回的完整性，减少漏项
- 扩充专题深审范围，覆盖真实业务审查场景
- 增强最终汇总的对比、聚类、找漏与解释能力
- 让结果不仅“能输出”，还要“可解释、可追踪、可复核”


## 2. 成熟版总体架构

成熟版 V2 建议采用“四层架构”：

1. 全文基线审查层
2. 结构识别与证据召回层
3. 专题深审层
4. 对比汇总与统一报告层

其中：

- 第一层负责从全文视角先抓通用风险
- 第二层负责为各专题建立高召回证据池
- 第三层负责按专题做定向深审
- 第四层负责风险聚类、差异对比、漏项分析和统一输出


## 3. 四层详细设计

### 3.1 第一层：全文基线审查层

#### 作用

- 从全文视角抓通用合规风险
- 输出第一版全局审查结果
- 作为专题结果对照基线

#### 输入

- 招标文件全文

#### 输出

- `baseline_report`
- `baseline_risks`
- `baseline_summary`

#### 擅长识别的问题

- 资格条件限制竞争
- 评分办法不量化
- 商务条款失衡
- 明显品牌倾向
- 验收、付款、履约安排不合理

#### 保留原因

全文主审仍然必要，因为它提供全局视角，可以帮助后续：

- 判断专题是否漏掉明显问题
- 形成后续汇总比对基线
- 给出对整份文件的第一印象风险图谱


### 3.2 第二层：结构识别与证据召回层

这是成熟版 V2 的核心升级重点。

#### 当前问题

当前结构层主要依赖：

- 标题规则
- 编号规则
- 关键词命中
- 简单模块打分

因此存在以下不足：

- 切分过碎
- 模块识别偏粗
- 跨章节证据容易漏召回
- 程序性模板条款容易误入专题
- 某些专题只召回到局部片段，无法覆盖完整上下文

#### 成熟版目标

结构层不再只是“章节切分”，而是升级为“证据召回层”。

#### 时延与成本边界

成熟版第二层虽然允许引入 LLM 做模块识别，但必须受成本和时延约束：

- 单次审查第二层最多触发 `1` 次结构识别 LLM 调用
- 仅当规则识别的模块置信度不足时才触发该调用
- 若全文长度低于 `20,000` 字且规则识别覆盖率达到阈值，则跳过 LLM 结构识别
- 第二层单独超时上限建议为 `90` 秒
- 超时或失败时必须退化到“规则切分 + 规则模块映射”模式，不能阻断全链路

建议在运行产物中记录：

- `structure_llm_used`
- `structure_llm_latency_ms`
- `structure_fallback_used`
- `section_count_before_merge`
- `section_count_after_merge`

#### 推荐处理流程

1. 弱结构切分
2. 模块概率识别
3. 专题证据召回
4. 二次找漏与聚类

#### 具体说明

##### 1. 弱结构切分

按以下线索将全文切成候选段落：

- 章节标题
- 编号结构
- 表格块
- 段落密度
- 关键词密集度

输出：

- `SectionCandidate[]`

##### 2. 模块概率识别

对每个候选段落识别它更可能属于哪个模块：

- 资格条件
- 评分办法
- 技术要求
- 商务条款
- 验收条款
- 程序条款
- 政策条款

建议方式：

- 规则模型先打粗标签
- LLM 做二次模块识别

输出：

- `ModuleHit[]`
- `module_map`

##### 3. 专题证据召回

围绕每个专题，从不同模块中召回证据片段，而不是只取单个章节。

例如“付款条款失衡”专题，可能需要同时召回：

- 商务要求中的付款节点
- 验收条款中的付款触发条件
- 合同专用条款中的结算与保证金

输出：

- `EvidenceBundle[]`
- `topic_evidence_map`

##### 4. 二次找漏与聚类

在第一次召回完成后，基于专题问题做“二次找漏”：

- 找遗漏章节
- 找跨章节关联条款
- 找前后矛盾的表达

输出：

- `topic_evidence_bundles`
- `missing_evidence_hints`


### 3.3 第三层：专题深审层

成熟版不建议只保留 4 个专题，建议扩展为 8 到 10 个专题。

#### 推荐专题列表

1. `qualification`
   供应商资格条件

2. `performance_staff`
   业绩、证书、奖项、人员要求

3. `scoring`
   评分办法与评分项

4. `samples_demo`
   样品、演示、答辩要求

5. `technical_bias`
   技术参数倾向性、品牌型号、专利专有技术

6. `technical_standard`
   标准引用、标准有效性、名称编号一致性、检测认证要求

7. `contract_payment`
   付款、履约期限、保证金、违约责任

8. `acceptance`
   验收主体、验收方式、验收标准

9. `procedure`
   质疑、澄清、解释权、程序门槛

10. `policy`
    中小企业、节能环保、政策条款

#### 每个专题的输入

- 对应专题的 `EvidenceBundle`
- 全文摘要信息
- 基线层输出的相关提示

#### 每个专题的输出

- `risk_points`
- `summary`
- `coverage_note`
- `missing_evidence`
- `need_manual_review`

#### 专题层的成熟版要求

- 每个专题只看相关证据，不直接吞全文
- 每个专题都要输出“覆盖说明”
- 若关键证据不足，要明确写“需人工复核”
- 若专题发现了基线层未发现的问题，要保留来源标记

#### 专题边界契约

为降低专题重叠带来的重复命中和冲突合并复杂度，成熟版必须先定义专题边界。

建议采用如下主归属规则：

- `qualification`
  只处理法定资格、特定资格、信用、联合体、供应商主体资格
- `performance_staff`
  只处理业绩、证书、奖项、人员、社保、驻场，不处理法定资格
- `scoring`
  只处理评分办法、评分项、分值、量化标准，不处理资格准入
- `samples_demo`
  只处理样品、演示、答辩及其评审逻辑
- `technical_bias`
  只处理品牌、型号、专利、专有技术、参数倾向性
- `technical_standard`
  只处理标准号、标准名称、标准版本、检测报告、CMA/CNAS、数值一致性
- `contract_payment`
  只处理付款、保证金、履约期限、违约责任
- `acceptance`
  只处理验收主体、验收方式、验收标准、验收程序
- `procedure`
  只处理质疑、澄清、解释权、程序性受理门槛
- `policy`
  只处理中小企业、节能环保、政策适用条款

#### 风险合并契约

进入第四层之前，每个专题风险点至少要带以下签名字段：

- `topic`
- `title`
- `review_type`
- `source_locations[]`
- `source_excerpt_hash`
- `severity`
- `source_rule`
  取值建议：`baseline` / `topic` / `cross_check`

建议的合并优先级：

1. 完全相同原文位置 + 相同问题标题，直接合并
2. 不同专题但原文摘录哈希相同，合并为一个主风险点
3. 原文位置不同但问题标题、审查类型、核心判断高度相似，进入人工或规则聚类

严重级别冲突裁决规则：

- 默认取最高风险级别
- 若一个专题为“需人工复核”，另一个专题为明确高/中/低风险，则保留明确级别并在备注中记录冲突
- 若多个专题结论冲突且证据范围不同，则输出“冲突标记 + 需人工复核”


### 3.4 第四层：对比汇总与统一报告层

这是成熟版 V2 必须新增的能力层。

#### 当前问题

当前汇总主要是：

- 基线结果 + 专题结果去重合并

不足在于：

- 缺少冲突识别
- 缺少漏项分析
- 缺少跨专题风险聚类
- 缺少基线与专题的差异解释

#### 成熟版目标

汇总层要从“拼接器”升级为“分析器”。

#### 建议能力

##### 1. 重复风险聚类

当多个专题命中同一问题时，合并为一个主风险点。

例如：

- 技术倾向性专题发现“品牌指定”
- 样品专题也发现“品牌指定”

最终应聚成一个主风险点，并在详情中保留多个证据来源。

##### 2. 冲突识别

识别这类情况：

- 全文主审说“未发现明显问题”
- 专题深审说“存在高风险”

或：

- 一个专题判断为低风险
- 另一个专题判断为高风险

输出：

- `conflicts`

##### 3. 漏项分析

识别这类情况：

- 某专题未召回关键章节
- 某专题证据不足但仍给了结论
- 全文主审与专题覆盖区域差异过大

输出：

- `coverage_summary`
- `missing_topic_coverage`

##### 4. 来源标识

每个最终风险点应标记来源：

- `baseline`
- `topic`
- `cross-check`

##### 5. 统一主风险点编号

最终报告中的风险点编号必须统一，不应保留多个专题重复编号。

输出：

- `final_risks`
- `comparison_summary`
- `manual_review_items`
- `final_report`

#### 调用预算与降级策略

成熟版第四层虽然需要做比对分析，但必须约束调用预算，不能无限增加链路复杂度。

建议的单次审查最大调用预算：

- 第一层全文基线审查：`1` 次
- 第二层结构识别：`0-1` 次
- 第三层专题深审：默认 `6` 次，最多 `8` 次
- 第四层对比汇总：默认 `0-1` 次

推荐总预算：

- 默认模式：`7` 次以内
- 加强模式：`10` 次以内

推荐降级顺序：

1. 保留第一层
2. 保留高价值专题：
   `technical_standard`、`scoring`、`contract_payment`、`qualification`
3. 跳过低优先级专题：
   `policy`、`procedure`
4. 第四层对比汇总改为规则聚类，不再额外调用 LLM

推荐超时边界：

- 单专题超时：`60-90` 秒
- 总任务超时：`12-15` 分钟

推荐缓存策略：

- 相同 `run_id` 内结构识别结果只生成一次
- 同一文件内容哈希相同且配置未变时，可复用 `document_map.json`
- 专题 evidence bundle 生成后应持久化，失败重试时不必重新召回


## 4. 重点业务能力补强

### 4.1 技术细节型风险识别

成熟版 V2 需要重点加强以下能力：

- 标准与采购标的相关性判断
- 标准是否废止或版本滞后
- 标准名称与编号是否一致
- 强制性标准是否显著标识
- CMA/CNAS/检测报告要求是否歧义
- 数值、单位、数量、时间是否前后冲突

这是 V2 相比 V1 最重要的差异化能力之一。


### 4.2 跨章节证据聚合

成熟版必须能支持：

- 一个风险点对应多个章节
- 同类问题跨章节交叉印证
- 某项条款在“商务要求 + 验收条款 + 合同条款”中同时出现时的统一分析


### 4.3 可解释性

成熟版输出不只是最终结论，还应解释：

- 哪个专题发现了该问题
- 哪些原文支撑该问题
- 哪些章节未能召回或需补充
- 是否与基线层存在差异

- `comparison.json`
- `evidence_map.json`
- `coverage_summary.json`

### 兼容要求

- `document_map.json` 保持现有顶层字段不删，允许新增：
  - `module_hits`
  - `evidence_candidates`
  - `coverage_hints`
- `topic_reviews/*.json` 保持现有字段不删，允许新增：
  - `topic_version`
  - `risk_signature`
  - `missing_evidence`
  - `source_rule`
- `v2_overview.json` 保持现有列表结构不删，允许新增：
  - `coverage_rate`
  - `fallback_used`
  - `comparison_ready`

### Web 兼容策略

- 当前页面优先继续消费：
  - `review.md`
  - `topic_reviews/*.json`
  - `v2_overview.json`
- 新页面能力逐步接入：
  - 若存在 `comparison.json`，则展示差异与聚类结果
  - 若不存在，则页面自动退回当前展示模式

### 历史结果兼容策略

- 旧运行目录没有 `comparison.json` 时，不应报错
- 历史记录页应允许“回放旧结果”
- 所有下载接口对旧产物继续可用
- 新产物不存在时返回隐藏按钮，而不是返回错误页面

## 8. 分期建设建议

### 第一期：结构识别与证据层升级

目标：

- 把第二层从“规则切分层”升级成“证据召回层”

建设内容：

- 引入 `evidence.py`
- 规则切分保留
- 增加 LLM 模块识别
- 为每个专题生成 `evidence_bundle`
- 增加证据召回补漏逻辑


### 第二期：专题扩容

优先补充以下专题：

1. `technical_standard`
2. `acceptance`
3. `samples_demo`
4. `performance_staff`

目标：

- 从粗粒度专题升级到业务完整专题体系


### 第三期：对比汇总层建设

目标：

- 把最终汇总从“拼接合并”升级为“分析型汇总”

建设内容：

- 风险聚类
- 主审与专题差异比对
- 漏项与冲突识别
- 主风险点统一编号


### 第四期：页面成熟化

目标：

- 页面不仅能看结果，还能看覆盖、差异和证据链

建设内容：

- 专题覆盖度展示
- 基线/专题差异视图
- 风险来源标签
- 证据链展示


## 9. 成熟版验收标准

成熟版 V2 的验收不能只使用“明显提升”“不明显漏掉”这类主观表述，必须绑定固定样本集和量化指标。

### 固定回归样本集

建议建立一套最小验收样本集：

- `10` 份标准样本：
  - 资格限制类 `2` 份
  - 评分办法问题类 `2` 份
  - 技术标准/检测问题类 `2` 份
  - 商务验收付款类 `2` 份
  - 程序性条款类 `2` 份
- `5` 份干净样本：
  - 风险点较少或主要为低风险

每份样本需人工标注：

- 标准答案风险点列表
- 每个风险点所属专题
- 支撑原文位置
- 是否允许“需人工复核”

### 量化验收指标

成熟版至少应满足以下指标：

- 结构稳定性：
  同一文件连续运行 `3` 次，`document_map.json` 中模块主归属一致率不低于 `85%`
- 证据召回率：
  对人工标注的关键条款，专题 evidence bundle 召回率不低于 `90%`
- 专题命中率：
  对人工标注的高风险与中风险问题，专题层召回率不低于 `80%`
- 技术细节命中率：
  对“标准废止/版本滞后/名称编号不一致/CMA-CNAS歧义”类问题，命中率不低于 `70%`
- 重复风险合并率：
  多专题重复命中问题的聚类合并准确率不低于 `80%`
- 人工复核率上限：
  在标准样本集中，`need_manual_review` 占最终风险点比例不高于 `35%`
- 链路时延：
  默认模式下，单次审查 `P95` 不超过 `12` 分钟
- LLM 调用预算：
  默认模式平均调用次数不高于 `7` 次

### 页面与产物验收项

- 最终汇总具备差异分析与风险聚类能力
- 页面可展示：
  - 最终报告
  - 分专题结果
  - 覆盖说明
  - 需人工复核原因

### 不达标处理

若上述任一核心指标未达标，则视为未完成成熟版验收，应明确落回：

- 结构层继续使用规则召回
- 降低专题数量
- 暂缓启用 comparison 结果进入页面主视图


## 10. 结论

成熟版 V2 的关键不是“多调几次模型”，而是三项升级：

1. 第二层从“规则切分”升级成“证据召回层”
2. 第三层从“少量粗专题”升级成“完整专题体系”
3. 第四层新增“对比汇总层”，补上差异分析与风险聚类能力

后续开发建议优先级：

1. 结构与证据层升级
2. 专题扩容
3. 对比汇总层建设
4. 页面成熟化
